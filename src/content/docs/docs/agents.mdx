Genkit Agents let you add AI-powered features - conversational assistants,
approval workflows, background report generators - to your application
with a single server definition and standard client helpers. Define a
model, system prompt, tools, and session state once; Genkit handles
message history, tool-calling loops, streaming, and persistence so you
can focus on your product. Under the hood, agents are powered by prompts
and the `generate()` API, so they have full access to
[middleware](/docs/middleware) - add retries, fallbacks, tool approval,
filesystem access, or your own custom hooks to any agent. Agents replace
the deprecated [`ai.chat()`](/docs/chat) API.

All of the code samples in this guide can be found in the [agents sample](https://github.com/genkit-ai/genkit/tree/main/js/testapps/agents) on GitHub.

## Before you begin

Before reading this page, you should be familiar with [generating content](/docs/models),
[tool calling](/docs/tool-calling), and [prompts](/docs/dotprompt).

:::note[Beta import]
All agent APIs are imported from **`genkit/beta`**, not `genkit`. This
applies to `genkit()`, session stores, and client-side types.
:::

```ts
import { genkit } from 'genkit/beta';
import { googleAI } from '@genkit-ai/google-genai';

const ai = genkit({
  plugins: [googleAI()],
  model: googleAI.model('gemini-flash-latest'),
});
```

## Defining agents

### `ai.defineAgent()`

The primary way to create an agent. Define a model, system prompt, tools, and
optionally a session store - all in one place:

```ts
const weatherAgent = ai.defineAgent({
  name: 'weatherAgent',
  model: 'googleai/gemini-flash-latest',
  system: 'You are a helpful weather assistant. Use the getWeather tool.',
  tools: [getWeather],
  store: new FileSessionStore('./.snapshots'),
});
```

**With typed custom state:**

Define your state shape with a Zod schema and pass it as `stateSchema`.
This gives you TypeScript types, runtime validation, and JSON Schema
for the developer UI:

```ts
const TaskStateSchema = z.object({
  tasks: z.array(z.object({
    id: z.number(),
    title: z.string(),
    done: z.boolean(),
  })),
  nextId: z.number(),
});

type TaskState = z.infer<typeof TaskStateSchema>;

const taskAgent = ai.defineAgent({
  name: 'taskAgent',
  stateSchema: TaskStateSchema,
  model: 'googleai/gemini-flash-latest',
  system: 'You are a task management assistant. Use the provided tools.',
  tools: [addTask, toggleTask, removeTask],
});
```

### `ai.definePromptAgent()`

Wire an agent from a [Dotprompt](/docs/dotprompt) `.prompt` file. Useful when
prompt authors (non-developers) edit templates separately from agent wiring:

```
---
model: googleai/gemini-flash-latest
tools:
  - getAttractions
  - getFlightInfo
---

{{role "system"}}
You are a friendly trip planning assistant. Help users plan trips by
suggesting attractions and looking up flight information.

{{history}}
```

```ts
const tripPlannerAgent = ai.definePromptAgent({
  promptName: 'tripPlanner',
  store: new FileSessionStore('./.snapshots'),
});
```

### `ai.defineCustomAgent()`

For full control over orchestration - multiple model calls, custom status
streaming, and manual session management:

```ts
const researchAgent = ai.defineCustomAgent(
  { name: 'researchAgent' },
  async (sess, { sendChunk }) => {
    let lastMessage;

    await sess.run(async (input) => {
      const userText =
        input.messages?.[input.messages.length - 1]?.content[0]?.text || '';

      // Step 1: Decompose the question
      sendChunk({ status: 'Decomposing question…' });
      const decompose = await ai.generate({
        model: 'googleai/gemini-flash-lite-latest',
        prompt: `Break this into 2-3 sub-questions as a JSON array: "${userText}"`,
        output: { format: 'json', schema: z.array(z.string()).min(2).max(3) },
      });
      const subQuestions = decompose.output ?? [userText];

      // Step 2: Research each sub-question
      const subAnswers = [];
      for (let i = 0; i < subQuestions.length; i++) {
        sendChunk({ status: `Researching (${i + 1}/${subQuestions.length})…` });
        const research = await ai.generate({
          prompt: `Answer concisely: ${subQuestions[i]}`,
        });
        subAnswers.push({ question: subQuestions[i], answer: research.text });
      }

      // Step 3: Synthesize - stream the final response
      sendChunk({ status: 'Synthesizing…' });
      const synthesisStream = ai.generateStream({
        prompt: `Synthesize these findings into a cohesive answer:\n${
          subAnswers.map((sa) => `Q: ${sa.question}\nA: ${sa.answer}`).join('\n\n')
        }`,
      });

      for await (const chunk of synthesisStream.stream) {
        sendChunk({ modelChunk: chunk });
      }

      lastMessage = (await synthesisStream.response).message;
      sess.addMessages([lastMessage]);
    });

    return {
      message: lastMessage || {
        role: 'model',
        content: [{ text: 'Research complete.' }],
      },
    };
  }
);
```

Use `defineCustomAgent` when you need:
- Multiple sequential model calls with logic between them
- Custom `status` streaming (not just model token chunks)
- Direct session manipulation (`addMessages`, `updateCustom`)
- Different models for different steps

## State management strategies

Every agent uses one of two strategies for maintaining session state across
turns. This choice affects how the client continues a conversation, and
**sending the wrong continuation token will throw an error** - so pick a
strategy before building your client.

| | Server-managed | Client-managed |
|---|---|---|
| **Setup** | Define the agent with a `store` | Omit `store` |
| **Continue a turn** | Send `snapshotId` | Send the full `state` blob |
| **Server stores state?** | Yes | No |
| **Supports branching?** | Yes | Manual (client manages copies of `state`) |
| **Supports background execution?** | Yes | No |

See [Session persistence](#session-persistence) for detailed examples of
each strategy, including session restoration, branching, and pruning.

## Serving agents over HTTP

Before your client app can talk to an agent, expose it as an HTTP endpoint.
Use `expressHandler` from `@genkit-ai/express`:

```ts
import { expressHandler } from '@genkit-ai/express';
import express from 'express';

const app = express();
app.use(express.json());

// Main agent endpoint - handles run, streaming, and multi-turn
app.post('/api/weatherAgent', expressHandler(weatherAgent));

// Snapshot endpoint - lets clients restore a session from a snapshotId
app.post('/api/weatherAgent/state', expressHandler(weatherAgent.getSnapshotDataAction));

app.listen(8080);
```

Every agent gets one primary endpoint. Agents that use a store also expose a
`getSnapshotDataAction` for session restoration, and agents that support
background execution expose an `abortAgentAction`.

On the **client side**, use `streamFlow()` and `runFlow()` from
`genkit/beta/client`, along with the shared types:

```ts
import type { AgentInit, AgentInput, AgentOutput, AgentStreamChunk } from 'genkit/beta';
import { streamFlow, runFlow } from 'genkit/beta/client';
```

Agents are designed as HTTP endpoints from the start - they slot into your
Express, Next.js, or Cloud Functions app like any other route. Your client
code is just `streamFlow()` with standard types - no agent-specific SDK,
no WebSocket setup, no custom protocol.

The rest of this page shows both the server and client sides of each pattern.

## Multi-turn streaming chat

The most common pattern: a streaming chat where the client sends messages,
receives streamed tokens and tool-call events, and maintains session
continuity across turns.

### Server

Use `agent.run()` with an `onChunk` callback for streaming:

```ts
const result = await weatherAgent.run(
  {
    messages: [{ role: 'user', content: [{ text: 'Weather in London?' }] }],
  },
  { onChunk: (chunk) => console.log(chunk) }
);

console.log(result.result.message);    // Model's response message
console.log(result.result.state);      // Session state (for next turn)
console.log(result.result.snapshotId); // Snapshot ID (if using a store)
```

Continue a conversation by passing state from the previous turn:

```ts
// Turn 2 - resume with state
const turn2 = await weatherAgent.run(
  {
    messages: [{ role: 'user', content: [{ text: 'What about Paris?' }] }],
  },
  { init: { state: turn1.result.state } }
);

// Or resume with snapshotId (requires a store)
const turn2Alt = await weatherAgent.run(
  {
    messages: [{ role: 'user', content: [{ text: 'What about Paris?' }] }],
  },
  { init: { snapshotId: turn1.result.snapshotId } }
);
```

### Client

Use `streamFlow()` to stream the agent's response. Each chunk can contain
model text, tool calls, and tool responses - giving your UI full visibility
into what the agent is doing:

```ts
// Build the request
const input: AgentInput = {
  messages: [{ role: 'user', content: [{ text: userMessage }] }],
};

// On subsequent turns, send back state or snapshotId from the previous turn
const init: AgentInit = previousState
  ? { state: previousState }
  : previousSnapshotId
    ? { snapshotId: previousSnapshotId }
    : {};

// Stream the response
const response = streamFlow<AgentOutput, AgentStreamChunk, AgentInit>({
  url: '/api/weatherAgent',
  input,
  init,
});

for await (const chunk of response.stream) {
  if (chunk.modelChunk) {
    for (const part of chunk.modelChunk.content || []) {
      if (part.text) appendToUI(part.text);           // Stream text to UI
      if (part.toolRequest) showToolCall(part.toolRequest);   // Show tool invocation
      if (part.toolResponse) showToolResult(part.toolResponse); // Show tool result
    }
  }
}

// Read the final result
const result = await response.output;

// Save for the next turn
previousState = result.state;
previousSnapshotId = result.snapshotId;
```

For non-streaming requests, use `runFlow()`:

```ts
const result = await runFlow<AgentOutput, AgentInit>({
  url: '/api/weatherAgent',
  input: {
    messages: [{ role: 'user', content: [{ text: 'Weather in London?' }] }],
  },
});
```

## Session persistence

This section covers each [state management strategy](#state-management-strategies)
in detail - stores, session restoration, client-managed state, branching, and
pruning.

### Server-side stores

Use a store to persist session state across requests. Genkit provides
`FileSessionStore` and `InMemorySessionStore`:

```ts
import { FileSessionStore, InMemorySessionStore } from 'genkit/beta';

// File-based persistence
const agent = ai.defineAgent({
  name: 'myAgent',
  // ...
  store: new FileSessionStore('./.snapshots'),
});

// In-memory (for development/testing)
const agent = ai.defineAgent({
  name: 'myAgent',
  // ...
  store: new InMemorySessionStore(),
});
```

With a store, each turn returns a `snapshotId` that you can use to restore
the session later - for example, when a user bookmarks a chat URL or
returns to a page.

**Server - retrieve snapshot data:**

```ts
const snapshot = await agent.getSnapshotDataAction(snapshotId);
// snapshot.state.messages - full message history
// snapshot.state.custom - custom state
// snapshot.state.artifacts - artifacts
```

**Client - restore a session on page load:**

Expose the `getSnapshotDataAction` as an endpoint, then call it from the
client to rebuild the conversation UI:

```ts
// Server
app.post('/api/myAgent/state', expressHandler(agent.getSnapshotDataAction));
```

```ts
// Client - restore from a snapshotId (e.g., from the URL)
const snapshot = await runFlow({
  url: '/api/myAgent/state',
  input: snapshotId,
});

// Rebuild the chat UI from the message history
for (const msg of snapshot.state.messages) {
  for (const part of msg.content) {
    if (part.text) renderMessage(msg.role, part.text);
  }
}
```

This lets you store the `snapshotId` in the URL, a cookie, or a database,
and fully restore the conversation later.

### Client-managed state

If you don't provide a `store`, the agent is stateless on the server. The
client must round-trip the full `state` blob on every turn:

```ts
// Server - no store
const clientAgent = ai.defineAgent({
  name: 'clientAgent',
  model: 'googleai/gemini-flash-latest',
  system: 'You are a helpful assistant.',
  tools: [getWeather],
  // No `store` property
});
```

```ts
// Client - round-trip the state blob
const response = streamFlow<AgentOutput, AgentStreamChunk, AgentInit>({
  url: '/api/clientAgent',
  input: {
    messages: [{ role: 'user', content: [{ text: userMessage }] }],
  },
  // On the first turn, init is empty. On subsequent turns, send back the
  // state blob from the previous response.
  init: previousState ? { state: previousState } : {},
});

for await (const chunk of response.stream) {
  // ... render streamed content
}

const result = await response.output;
// Store the state blob for the next turn
previousState = result.state;
```

The trade-off: no server storage needed, but the client sends the full
message history on every request. This works well for short conversations
or when you want a fully stateless server.

### Session branching

With a store, you can branch conversations from any snapshot point - creating
parallel conversation threads from the same starting point:

```ts
const turn1 = await agent.run({
  messages: [{ role: 'user', content: [{ text: 'Hello!' }] }],
});
const branchPoint = turn1.result.snapshotId;

// Branch A
const branchA = await agent.run(
  { messages: [{ role: 'user', content: [{ text: 'My name is Alice.' }] }] },
  { init: { snapshotId: branchPoint } }
);

// Branch B - from the same point
const branchB = await agent.run(
  { messages: [{ role: 'user', content: [{ text: 'My name is Bob.' }] }] },
  { init: { snapshotId: branchPoint } }
);
```

### Snapshot chain pruning

Control how many snapshots are retained in the chain with
`maxPersistedChainLength`:

```ts
const store = new FileSessionStore('./.snapshots', {
  maxPersistedChainLength: 3, // Keep only the last 3 snapshots
});
```

## Custom state and artifacts

### Custom state

Agents don't just produce text - they produce **structured app state**
that drives your UI. Custom state is useful when your application needs a
reliable source of truth that the model can't maintain in chat history
alone - exact IDs, counters, structured records, or any data where
precision matters. Tools can read and mutate typed custom state via
`ai.currentSession()`:

```ts
const addTask = ai.defineTool(
  {
    name: 'addTask',
    description: 'Add a new task to the task list.',
    inputSchema: z.object({ title: z.string() }),
    outputSchema: z.object({ id: z.number(), title: z.string(), done: z.boolean() }),
  },
  async (input) => {
    const session = ai.currentSession<TaskState>();
    let newTask!: TaskItem;
    session.updateCustom((state) => {
      const s = state || { tasks: [], nextId: 1 };
      newTask = { id: s.nextId, title: input.title, done: false };
      s.tasks.push(newTask);
      s.nextId++;
      return s;
    });
    return newTask;
  }
);
```

Access custom state from the result:

```ts
const result = await taskAgent.run(
  { messages: [{ role: 'user', content: [{ text: 'Add: buy groceries' }] }] },
  {
    init: {
      state: {
        custom: { tasks: [], nextId: 1 },
        messages: [],
        artifacts: [],
      },
    },
  }
);

console.log(result.result.state?.custom); // { tasks: [...], nextId: 2 }
```

On the client, custom state is available in `result.state.custom` - use it
to render application-specific UI alongside the chat. For example, render
a task list driven entirely by agent state:

```ts
// Client - render a task list from custom state
const tasks = result.state?.custom?.tasks ?? [];
for (const task of tasks) {
  renderTaskItem(task.id, task.title, task.done);
}
```

### Artifacts

Tools can emit artifacts (generated files, code, etc.) via
`ai.currentSession().addArtifacts()`:

```ts
const emitArtifact = ai.defineTool(
  {
    name: 'emitArtifact',
    description: 'Emit a generated code file to the workspace.',
    inputSchema: z.object({ name: z.string(), content: z.string() }),
    outputSchema: z.object({ status: z.string() }),
  },
  async (input) => {
    ai.currentSession().addArtifacts([{
      name: input.name,
      parts: [{ text: input.content }],
    }]);
    return { status: `Artifact ${input.name} emitted.` };
  }
);
```

Artifacts are returned in `result.artifacts` on the client and also persisted
in the session state.

## Agents with interrupts

Agents work seamlessly with [interrupts](/docs/interrupts) for human-in-the-loop
workflows - approvals, confirmations, or any time you need user input
mid-flow.

### Server

Define an interrupt and include it in the agent's tools:

```ts
const userApproval = ai.defineInterrupt({
  name: 'userApproval',
  description: 'Ask the user for approval before proceeding.',
  inputSchema: z.object({
    action: z.string(),
    details: z.string(),
  }),
  outputSchema: z.object({
    approved: z.boolean(),
    feedback: z.string().optional(),
  }),
});

const bankingAgent = ai.defineAgent({
  name: 'bankingAgent',
  model: 'googleai/gemini-flash-latest',
  system: 'You are a banking assistant. Always use userApproval before transfers.',
  tools: [userApproval, transferMoney],
  store: new InMemorySessionStore(),
});
```

### Client

The interrupt pattern has two phases: **detect** the interrupt in the
response, then **resume** by sending a tool response.

**1. Send the initial request and detect the interrupt:**

```ts
const response = streamFlow<AgentOutput, AgentStreamChunk, AgentInit>({
  url: '/api/bankingAgent',
  input: {
    messages: [{ role: 'user', content: [{ text: 'Transfer $500 to savings' }] }],
  },
  init: previousSnapshotId ? { snapshotId: previousSnapshotId } : {},
});

for await (const chunk of response.stream) {
  // ... render streamed content
}

const result = await response.output;

// Check if the agent paused for approval
const approvalRequest = result.message?.content.find(
  (p) => p.toolRequest?.name === 'userApproval'
);

if (approvalRequest?.toolRequest) {
  // Show an approval dialog in the UI
  // Save result.snapshotId and the toolRequest ref for resumption
}
```

**2. Resume after the user responds:**

```ts
// User clicked "Approve" - send a tool response to resume the flow
const resumed = streamFlow<AgentOutput, AgentStreamChunk, AgentInit>({
  url: '/api/bankingAgent',
  input: {
    messages: [{
      role: 'tool',
      content: [{
        toolResponse: {
          name: 'userApproval',
          ref: savedToolRequestRef,
          output: { approved: true, feedback: 'Looks good' },
        },
      }],
    }],
  },
  // Resume from the exact snapshot where the flow paused
  init: { snapshotId: savedSnapshotId },
});

for await (const chunk of resumed.stream) {
  // ... render the model's confirmation
}
```

The interrupt pattern uses tool calls as control flow - `userApproval`
never executes server-side. It exists solely to pause the agent and hand
control back to the client.

## Background execution

For long-running tasks (report generation, deep research), run the agent
in the background. The client submits a request, receives a `snapshotId`
immediately, and polls for the result.

### Server

Define the agent with a store (required for background execution) and
expose the state and abort endpoints:

```ts
const backgroundAgent = ai.defineAgent({
  name: 'reportAgent',
  model: 'googleai/gemini-flash-latest',
  system: 'You are a research analyst. Write comprehensive reports.',
  store: new InMemorySessionStore(), // Required for detach
});
```

```ts
app.post('/api/backgroundAgent', expressHandler(backgroundAgent));
app.post('/api/backgroundAgent/state', expressHandler(backgroundAgent.getSnapshotDataAction));
app.post('/api/backgroundAgent/abort', expressHandler(backgroundAgent.abortAgentAction));
```

### Client

**1. Submit with `detach: true`** - the server returns immediately:

```ts
const result = await runFlow<AgentOutput, AgentInit>({
  url: '/api/backgroundAgent',
  input: {
    messages: [{ role: 'user', content: [{ text: 'Write a report on AI trends' }] }],
    detach: true,
  },
  init: {},
});

const snapshotId = result.snapshotId; // Save for polling
```

**2. Poll the `/state` endpoint** until the status is `done`:

```ts
const poll = setInterval(async () => {
  const snapshot = await runFlow({
    url: '/api/backgroundAgent/state',
    input: snapshotId,
  });

  // snapshot.status: 'pending' | 'done' | 'failed' | 'aborted'
  if (snapshot.status === 'done') {
    clearInterval(poll);
    // Extract the report from the snapshot's message history
    const messages = snapshot.state.messages;
    const modelReply = messages.filter((m) => m.role === 'model').pop();
    displayReport(modelReply);
  }
}, 2000);
```

**3. Abort if needed:**

```ts
await runFlow({ url: '/api/backgroundAgent/abort', input: snapshotId });
```

## Multi-agent delegation

Use the `agents()` middleware from `@genkit-ai/middleware` to let an
orchestrator agent delegate tasks to specialized sub-agents.

### Server

```ts
import { agents } from '@genkit-ai/middleware';

// Define specialized sub-agents
const researcher = ai.defineAgent({
  name: 'researcher',
  model: 'googleai/gemini-flash-latest',
  system: 'You are a thorough research assistant. Use getWebResults to find information.',
  tools: [getWebResults],
});

const coder = ai.defineAgent({
  name: 'coder',
  model: 'googleai/gemini-flash-latest',
  system: 'You are an expert programmer. Write clean, well-commented code.',
});

// Orchestrator delegates to sub-agents via the `agents` middleware
const orchestratorAgent = ai.defineAgent({
  name: 'orchestrator',
  model: 'googleai/gemini-flash-latest',
  system: `You are a project assistant with access to sub-agents:
- **researcher**: research questions, fact-finding
- **coder**: writing code, debugging

Delegate to the appropriate sub-agent using the call_agent tool.`,
  use: [agents({ agents: ['researcher', 'coder'] })],
});
```

The middleware injects a `call_agent` tool. When the orchestrator model
calls it, the middleware runs the sub-agent and returns the response
as the tool result.

### Client

The client consumes the orchestrator exactly like any other agent. The
`call_agent` tool calls appear in the stream, so you can show delegation
status in the UI:

```ts
const response = streamFlow<AgentOutput, AgentStreamChunk, AgentInit>({
  url: '/api/orchestratorAgent',
  input: {
    messages: [{
      role: 'user',
      content: [{ text: 'Research sorting algorithms then write a quicksort in TypeScript.' }],
    }],
  },
  init: previousState ? { state: previousState } : {},
});

for await (const chunk of response.stream) {
  if (chunk.modelChunk) {
    for (const part of chunk.modelChunk.content || []) {
      if (part.text) appendToUI(part.text);
      if (part.toolRequest?.name === 'call_agent') {
        // Show which sub-agent is being called
        showStatus(`Delegating to ${part.toolRequest.input.agent}…`);
      }
      if (part.toolResponse?.name === 'call_agent') {
        showStatus(`Sub-agent responded`);
      }
    }
  }
}

const result = await response.output;
previousState = result.state;
```

## Bidirectional streaming

`agent.streamBidi()` creates a bidirectional streaming session - you can send
messages and receive streamed responses concurrently. This is useful for
server-side scripts or advanced scenarios where you need to queue multiple
messages before reading responses:

```ts
const session = weatherAgent.streamBidi({});

// Send messages
session.send({
  messages: [{ role: 'user', content: [{ text: 'Weather in Paris?' }] }],
});
session.send({
  messages: [{ role: 'user', content: [{ text: 'Now say that in French' }] }],
});
session.close(); // Signal no more input

// Read streamed responses
for await (const chunk of session.stream) {
  if (chunk.modelChunk) {
    for (const part of chunk.modelChunk.content || []) {
      if (part.text) process.stdout.write(part.text);
    }
  }
}

// Get the final result
const output = await session.output;
```

Resume a `streamBidi` session from a snapshot:

```ts
const session = weatherAgent.streamBidi({ snapshotId: previousSnapshotId });
```
