Genkit Agents let you add AI-powered features - conversational assistants,
approval workflows, background report generators - to your application
with a single server definition and a unified client API. Define a
model, system prompt, tools, and session state once; Genkit handles
message history, tool-calling loops, streaming, and persistence so you
can focus on your product. Under the hood, agents are powered by prompts
and the `generate()` API, so they have full access to
[middleware](/docs/middleware) - add retries, fallbacks, tool approval,
filesystem access, or your own custom hooks to any agent. Agents replace
the deprecated [`ai.chat()`](/docs/chat) API.

Agents are built around a **unified API**. The object returned by
`ai.defineAgent()` on the server and the one returned by `remoteAgent()` -
the typed HTTP client you create to call an agent from your app - expose the
*exact same interface*. You call `.chat()` to start a conversation,
`.sendStream()` to stream a turn, and read `.response` for the result -
whether you're calling an agent in-process or over HTTP. No agent-specific
SDK, no custom protocol, no manual state threading.

All of the code samples in this guide can be found in the [agents sample](https://github.com/genkit-ai/genkit/tree/main/js/testapps/agents) on GitHub.

## Before you begin

Before reading this page, you should be familiar with [generating content](/docs/models),
[tool calling](/docs/tool-calling), and [prompts](/docs/dotprompt).

:::note[Beta import]
All agent APIs are imported from **`genkit/beta`** (server) and
**`genkit/beta/client`** (client), not `genkit`. This applies to
`genkit()`, session stores, `remoteAgent()`, and the shared agent types.
:::

```ts
import { genkit, z } from 'genkit/beta';
import { googleAI } from '@genkit-ai/google-genai';

const ai = genkit({
  plugins: [googleAI()],
  model: googleAI.model('gemini-flash-latest'),
});
```

## Defining agents

Genkit offers three ways to define an agent. Use this quick guide to pick one:

- **`ai.defineAgent()`** - the right choice for most agents. Define a model,
  system prompt, and tools inline; Genkit runs the model and tool-calling
  loop for you.
- **`ai.definePromptAgent()`** - same as `defineAgent`, but the prompt lives
  in a [Dotprompt](/docs/dotprompt) `.prompt` file instead of inline. Use it
  when prompts are authored/versioned separately from code.
- **`ai.defineCustomAgent()`** - for full control of the orchestration logic
  (multiple model calls per turn, custom progress streaming, direct session
  manipulation). Use it only when `defineAgent` isn't flexible enough.

### `ai.defineAgent()`

The primary way to create an agent. Define a model, system prompt, tools, and
optionally a **session store** - all in one place. A session store is the
persistence backend that saves a conversation's message history and state
between requests, so a session can be resumed later (see
[Session persistence](#session-persistence)):

```ts
import { FileSessionStore } from 'genkit/beta';

const weatherAgent = ai.defineAgent({
  name: 'weatherAgent',
  system: 'You are a helpful weather assistant. Answer weather queries.',
  tools: [getWeather],
  // Persist session state between requests (see "Session persistence" section)
  store: new FileSessionStore('./.snapshots'),
});
```

The `model` is optional - it defaults to the model configured in `genkit()`.
Pass `model` to override it per-agent.

**With typed custom state:**

Define your state shape with a Zod schema and pass it as `stateSchema`.
This gives you TypeScript types and runtime validation. Genkit also
generates JSON Schema from the Zod schema and uses it in the developer
UI, where you can view and edit custom state directly while testing
your agent:

```ts
const TaskStateSchema = z.object({
  tasks: z.array(z.object({
    id: z.number(),
    title: z.string(),
    done: z.boolean(),
  })),
  nextId: z.number(),
});

type TaskState = z.infer<typeof TaskStateSchema>;

const taskAgent = ai.defineAgent({
  name: 'taskAgent',
  stateSchema: TaskStateSchema,
  system: 'You are a task management assistant. Use the provided tools.',
  tools: [addTask, toggleTask, removeTask],
});
```

### `ai.definePromptAgent()`

Wire an agent from a [Dotprompt](/docs/dotprompt) `.prompt` file. This
separates the prompt template from agent wiring, which is useful when:

- Non-developer prompt authors edit templates independently
- You want versioned, reviewable prompt files in source control
- Content teams manage prompt copy without touching application code
- You need to A/B test different prompt variants

```handlebars
---
model: googleai/gemini-flash-latest
tools:
  - getAttractions
  - getFlightInfo
---

{{role "system"}}
You are a friendly trip planning assistant. Help users plan trips by
suggesting attractions and looking up flight information.

{{history}}
```

```ts
const tripPlannerAgent = ai.definePromptAgent({
  promptName: 'tripPlanner',
  store: new FileSessionStore('./.snapshots'),
});
```

### `ai.defineCustomAgent()`

Use `defineCustomAgent` when you need full control over the agent's
orchestration logic. Unlike `defineAgent`, which runs a single model call
with automatic tool-calling loops, `defineCustomAgent` lets you:

- **Chain multiple model calls** with custom logic between them - for
  example, decompose a question, research each part, then synthesize
- **Stream custom status to the client** by mutating typed custom state -
  the runtime auto-emits the change so the UI can show progress like
  "Researching step 2 of 3…"
- **Manipulate the session directly** with `addMessages` and `updateCustom`,
  giving you control over exactly what gets persisted
- **Use different models for different steps** - a fast model for
  decomposition, a capable model for synthesis

The following example shows a research agent that decomposes a question,
researches each part, and streams status updates throughout. Notice that
status is streamed by mutating custom state with
`session.updateCustom()` - the runtime auto-emits a `customPatch` chunk
that the client receives as `chunk.custom`:

```ts
interface ResearchState {
  status?: string;
  subQuestions: string[];
  subAnswers: Array<{ question: string; answer: string }>;
}

const researchAgent = ai.defineCustomAgent(
  { name: 'researchAgent' },
  async (sess, { sendChunk }) => {
    let lastMessage;
    const session = ai.currentSession<ResearchState>();

    // sess.run() handles loading and saving session state around your logic.
    await sess.run(async (input) => {
      // Extract the user's question from the latest message.
      const userText =
        input.messages?.[input.messages.length - 1]?.content[0]?.text || '';

      // Step 1: Use a fast model to decompose the question into sub-questions.
      // Mutating custom state auto-emits a `customPatch` chunk to the client.
      session.updateCustom((s) => ({ ...s!, status: 'Decomposing question…' }));
      const decompose = await ai.generate({
        model: 'googleai/gemini-flash-lite-latest',
        prompt: `Break this into 2-3 sub-questions as a JSON array: "${userText}"`,
        output: { format: 'json', schema: z.array(z.string()).min(2).max(3) },
      });
      const subQuestions = decompose.output ?? [userText];

      // Step 2: Research each sub-question with the default model.
      const subAnswers = [];
      for (let i = 0; i < subQuestions.length; i++) {
        session.updateCustom((s) => ({
          ...s!,
          status: `Researching (${i + 1}/${subQuestions.length})…`,
        }));
        const research = await ai.generate({
          prompt: `Answer concisely: ${subQuestions[i]}`,
        });
        subAnswers.push({ question: subQuestions[i], answer: research.text });
      }

      // Step 3: Synthesize findings - stream the final response token by token.
      session.updateCustom((s) => ({ ...s!, status: 'Synthesizing…' }));
      const synthesisStream = ai.generateStream({
        prompt: `Synthesize these findings into a cohesive answer:\n${
          subAnswers.map((sa) => `Q: ${sa.question}\nA: ${sa.answer}`).join('\n\n')
        }`,
      });

      // Forward each model chunk to the client as it arrives.
      for await (const chunk of synthesisStream.stream) {
        sendChunk({ modelChunk: chunk });
      }

      // Persist the final model message in the session history.
      lastMessage = (await synthesisStream.response).message;
      sess.addMessages([lastMessage]);
    });

    return {
      message: lastMessage || {
        role: 'model',
        content: [{ text: 'Research complete.' }],
      },
    };
  }
);
```

A custom agent's `sendChunk` only accepts the agent stream-chunk shape
(`modelChunk`, `customPatch`, `artifact`, `turnEnd`). To stream
human-readable progress, mutate custom state with `session.updateCustom()`
as shown above rather than sending an ad-hoc status field.

## The unified agent API

Every agent - whether defined in-process with `ai.defineAgent()` or
accessed over HTTP from a client (via `remoteAgent()`, covered in
[Serving agents over HTTP](#serving-agents-over-http)) - exposes the same
surface. Once you have an agent, you start a `chat` and send turns:

```ts
// `weatherAgent` is the object returned by ai.defineAgent(...)
const chat = weatherAgent.chat();

// Streaming turn
const turn = chat.sendStream('What is the weather in Tokyo?');
for await (const chunk of turn.stream) {
  if (chunk.text) process.stdout.write(chunk.text); // streamed model text
}
const res = await turn.response;
console.log(res.text);        // final text
console.log(res.snapshotId);  // snapshot id (server-managed agents)
console.log(res.state);       // custom state

// Non-streaming follow-up turn - state is carried forward automatically.
const res2 = await chat.send('What about Paris?');
```

The chat tracks session continuity for you - no manual `snapshotId`/`state`
threading. The client-side `remoteAgent()` (below) returns the same object,
so client code looks identical (see [Serving agents over HTTP](#serving-agents-over-http)).

The core types:

| Type | What it is | Key members |
|---|---|---|
| `AgentAPI` | The agent itself | `chat()`, `loadChat()`, `getSnapshot()`, `abort()` |
| `AgentChat` | A stateful conversation | `send()`, `sendStream()`, `resume()`, `resumeStream()`, `detach()`, `abort()`, `snapshotId`, `state`, `messages`, `artifacts` |
| `AgentTurn` | One in-flight turn | `stream`, `response`, `abort()` |
| `AgentResponse` | A completed turn | `text`, `message`, `messages`, `toolRequests`, `interrupts`, `finishReason`, `snapshotId`, `state`, `artifacts`, `raw` |
| `AgentChunk` | A streamed chunk | `text`, `accumulatedText`, `reasoning`, `toolRequests`, `artifact`, `custom`, `raw` |
| `AgentInterrupt` | A paused tool request | `name`, `input`, `respond(output)`, `restart()` |
| `DetachedTask` | A background task handle | `snapshotId`, `poll()`, `wait()`, `abort()` |

`res.state` returns the **custom** state directly, while `res.raw.state`
holds the full `SessionState` (messages + custom + artifacts).

## State management strategies

Every agent uses one of two strategies for maintaining session state across
turns. This choice affects how the client continues a conversation - so pick
a strategy before building your client.

| | Server-managed | Client-managed |
|---|---|---|
| **Setup** | Define the agent with a `store` | Omit `store` |
| **How a turn continues** | The chat tracks `snapshotId`; resume with a `snapshotId` or `sessionId` | The chat tracks and round-trips the full `state` blob |
| **Server stores state?** | Yes | No |
| **Supports branching?** | Yes | Manual (open multiple chats from the same `state`) |
| **Supports background execution?** | Yes | No |

In both cases the `chat` object carries continuity automatically; the
difference is *where* the state lives. See
[Session persistence](#session-persistence) for detailed examples of each
strategy, including session restoration, branching, and pruning.

## Serving agents over HTTP

Before your client app can talk to an agent, expose it as an HTTP endpoint.
Use `expressHandler` from `@genkit-ai/express`:

```ts
import { expressHandler } from '@genkit-ai/express';
import express from 'express';

const app = express();
app.use(express.json());

// Main agent endpoint - handles run, streaming, and multi-turn.
// This is the only endpoint required for every agent.
app.post('/api/weatherAgent', expressHandler(weatherAgent));

// Optional: snapshot endpoint - only for server-managed (persisted) agents.
// Enables clients to restore/inspect a session via loadChat()/getSnapshot().
app.post('/api/weatherAgent/state', expressHandler(weatherAgent.getSnapshotDataAction));

// Optional: abort endpoint - only needed for background (detached) execution,
// so clients can cancel an in-flight background task.
app.post('/api/weatherAgent/abort', expressHandler(weatherAgent.abortAgentAction));

app.listen(8080);
```

Every agent exposes one **primary endpoint** - that's all you need for
streaming chat and multi-turn conversations. The other two endpoints are
optional and only matter in specific cases:

- **`getSnapshotDataAction`** (the `/state` endpoint) - expose this only for
  **server-managed agents** (those defined with a `store`). It lets clients
  restore or inspect a persisted session via `loadChat()` / `getSnapshot()`.
  Client-managed (store-less) agents don't need it - the client already holds
  the full state.
- **`abortAgentAction`** (the `/abort` endpoint) - expose this only when you
  use [background execution](#background-execution), so clients can cancel an
  in-flight detached task. Regular foreground turns are aborted client-side
  (see `AgentTurn.abort()`) and don't require this endpoint.

Both `getSnapshotDataAction` and `abortAgentAction` are **properties on the
agent object** returned by `ai.defineAgent()` (server-side only) - they're
part of the agent's serving surface, not the shared `AgentAPI` interface
that mirrors across server and client.

On the **client side**, create a typed client with `remoteAgent()` from
`genkit/beta/client`:

```ts
import { remoteAgent } from 'genkit/beta/client';

const agent = remoteAgent({ url: '/api/weatherAgent' });

const chat = agent.chat();
const res = await chat.send('Weather in Tokyo?');
console.log(res.text);
```

`remoteAgent()` defaults the state and abort URLs to `${url}/state` and
`${url}/abort`. Override them (or add request headers) if your routes differ:

```ts
const agent = remoteAgent({
  url: '/api/weatherAgent',
  stateUrl: '/api/weatherAgent/state',   // optional
  abortUrl: '/api/weatherAgent/abort',   // optional
  headers: { Authorization: `Bearer ${token}` }, // static or async function
});
```

Because `remoteAgent()` returns the same `AgentAPI` as the in-process agent,
**your client code is the same code you'd write on the server** - just
pointed at a URL. The rest of this page shows both sides of each pattern.

## Multi-turn streaming chat

The most common pattern: a streaming chat where the client sends messages,
receives streamed tokens and tool-call events, and maintains session
continuity across turns.

### Server

Use `chat.sendStream()` and iterate the stream, then await the response:

```ts
const chat = weatherAgent.chat();

const turn = chat.sendStream('Weather in London?');
for await (const chunk of turn.stream) {
  if (chunk.text) process.stdout.write(chunk.text);
}
const res = await turn.response;

console.log(res.message);    // Model's response message
console.log(res.state);      // Custom state
console.log(res.snapshotId); // Snapshot ID (if using a store)

// Continue the conversation - the same chat carries state forward.
const res2 = await chat.send('What about Paris?');
```

### Client

The client code mirrors the server. Use `chat.sendStream()`. Each chunk
exposes streamed text via `chunk.text`, and you can read tool calls and
responses off the raw model chunk:

```ts
import { remoteAgent } from 'genkit/beta/client';

const agent = remoteAgent({ url: '/api/weatherAgent' });
const chat = agent.chat();

const turn = chat.sendStream(userMessage);

for await (const chunk of turn.stream) {
  if (chunk.text) appendToUI(chunk.text); // streamed model text

  // Tool calls and responses arrive on the raw model chunk.
  for (const part of chunk.raw.modelChunk?.content ?? []) {
    if (part.toolRequest) showToolCall(part.toolRequest);
    if (part.toolResponse) showToolResult(part.toolResponse);
  }
}

const res = await turn.response;
// chat.snapshotId / chat.state are updated automatically for the next turn.
```

`chunk.accumulatedText` gives you the running text so far, which is handy
for rendering a growing message bubble.

For non-streaming requests, use `chat.send()`:

```ts
const res = await chat.send('Weather in London?');
console.log(res.text);
```

## Session persistence

This section covers each [state management strategy](#state-management-strategies)
in detail - stores, session restoration, client-managed state, branching, and
pruning.

### Server-side stores

Use a store to persist session state across requests. Genkit provides
`FileSessionStore` and `InMemorySessionStore`:

```ts
import { FileSessionStore, InMemorySessionStore } from 'genkit/beta';

// File-based persistence
const agent = ai.defineAgent({
  name: 'myAgent',
  // ...
  store: new FileSessionStore('./.snapshots'),
});

// In-memory (for development/testing)
const agent = ai.defineAgent({
  name: 'myAgent',
  // ...
  store: new InMemorySessionStore(),
});
```

With a store, each turn produces a `snapshotId` (available as
`chat.snapshotId` and `res.snapshotId`) that you can use to restore the
session later - for example, when a user bookmarks a chat URL or returns to
a page.

**Restore a chat with `loadChat()`:**

`loadChat()` reads a snapshot and returns a ready-to-use chat with history
restored. Accept either a `snapshotId` (an exact snapshot) or a `sessionId`
(the session's latest snapshot):

```ts
// Client - restore from a snapshotId (e.g., from the URL)
const chat = await agent.loadChat({ snapshotId });

// History is restored - rebuild the UI from chat.messages
for (const msg of chat.messages) {
  for (const part of msg.content) {
    if (part.text) renderMessage(msg.role, part.text);
  }
}

// Continue the conversation right where it left off.
const res = await chat.send('And what about tomorrow?');
```

**Read a snapshot without starting a chat:**

When you only need the data (not a live chat), use `getSnapshot()`:

```ts
const snapshot = await agent.getSnapshot(snapshotId);
// snapshot.state.messages  - full message history
// snapshot.state.custom    - custom state
// snapshot.state.artifacts - artifacts
// snapshot.status          - 'pending' | 'done' | 'failed' | 'aborted'
```

This lets you store the `snapshotId` in the URL, a cookie, or a database,
and fully restore the conversation later. The same `loadChat()` /
`getSnapshot()` methods work in-process on the server and over HTTP via
`remoteAgent()`.

### Client-managed state

If you don't provide a `store`, the agent is stateless on the server and the
client owns the session `state`. The `chat` object still tracks state for you
and round-trips it on every turn - the only difference is that there's no
`snapshotId`:

```ts
// Server - no store
const clientAgent = ai.defineAgent({
  name: 'clientAgent',
  system: 'You are a helpful assistant.',
  tools: [getWeather],
  // No `store` property
});
```

```ts
// Client - the chat round-trips the state blob automatically.
const agent = remoteAgent({ url: '/api/clientAgent' });
const chat = agent.chat();

await chat.send('Weather in Tokyo?');
await chat.send('What about Paris?'); // state carried forward automatically

// The full client-owned state blob:
console.log(chat.state);          // custom state
// res.raw.state holds the entire SessionState (messages + custom + artifacts)
```

The trade-off: no server storage needed, but the client holds (and the
transport sends) the full message history on every request. This works well
for short conversations or when you want a fully stateless server.

To seed initial custom state, pass it to `chat()`:

```ts
const chat = agent.chat({
  state: { custom: { tasks: [], nextId: 1 }, messages: [], artifacts: [] },
});
```

### Session branching

With a store, you can branch conversations from any snapshot point - creating
parallel conversation threads from the same starting point. Open multiple
chats from the same `snapshotId`:

```ts
const root = agent.chat();
const res1 = await root.send('Hello!');
const branchPoint = res1.snapshotId;

// Branch A
const branchA = agent.chat({ snapshotId: branchPoint });
await branchA.send('My name is Alice.');

// Branch B - forks from the SAME snapshot
const branchB = agent.chat({ snapshotId: branchPoint });
await branchB.send('My name is Bob.');
```

A `snapshotId` is an immutable checkpoint - like a git commit, the original
doesn't change when you branch from it. Each branch creates new, independent
snapshots. This works identically with `remoteAgent()` on the client, which
is how you can, for example, generate two response variants in parallel and
let the user pick one:

```ts
// Client - two parallel turns from the same branch point
const makeChat = () => agent.chat({ snapshotId });
const [a, b] = await Promise.all([
  makeChat().send(text),
  makeChat().send(text),
]);
// a.snapshotId !== b.snapshotId - pick one to continue from.
```

### Snapshot chain pruning

Control how many snapshots are retained in the chain with
`maxPersistedChainLength`:

```ts
const store = new FileSessionStore('./.snapshots', {
  maxPersistedChainLength: 3, // Keep only the last 3 snapshots
});
```

## Custom state and artifacts

### Custom state

Agents don't just produce text - they produce **structured app state**
that drives your UI. Custom state is useful when your application needs a
reliable source of truth that the model can't maintain in chat history
alone - exact IDs, counters, structured records, or any data where
precision matters. Tools can read and mutate typed custom state via
`ai.currentSession()` (reusing the `TaskState` schema from
[`ai.defineAgent()`](#aidefineagent) above).

To **mutate** state, call `session.updateCustom()` with a function that
receives the current state and returns the new state:

```ts
// Types inferred from the TaskStateSchema defined earlier.
type TaskState = z.infer<typeof TaskStateSchema>;
type TaskItem = TaskState['tasks'][number];

const addTask = ai.defineTool(
  {
    name: 'addTask',
    description: 'Add a new task to the task list.',
    inputSchema: z.object({ title: z.string() }),
    outputSchema: z.object({ id: z.number(), title: z.string(), done: z.boolean() }),
  },
  async (input) => {
    const session = ai.currentSession<TaskState>();
    let newTask!: TaskItem;
    session.updateCustom((state) => {
      // Normalize each field. On a fresh session custom state is an empty
      // object `{}` (see note below), so `state.tasks` may be undefined -
      // default per-field rather than relying on `state || defaults`.
      const tasks = state?.tasks ?? [];
      const nextId = state?.nextId ?? 1;
      newTask = { id: nextId, title: input.title, done: false };
      return { tasks: [...tasks, newTask], nextId: nextId + 1 };
    });
    return newTask;
  }
);
```

To **read** state without mutating it - for example in a "list tasks" tool -
call `session.getCustom()`, which returns the current custom state (or
`undefined` if nothing has been set yet):

```ts
const listTasks = ai.defineTool(
  {
    name: 'listTasks',
    description: 'List all tasks.',
    inputSchema: z.object({}),
    outputSchema: z.object({
      tasks: z.array(
        z.object({ id: z.number(), title: z.string(), done: z.boolean() })
      ),
    }),
  },
  async () => {
    const session = ai.currentSession<TaskState>();
    const state = session.getCustom();
    return { tasks: state?.tasks ?? [] };
  }
);
```

Access custom state from the response or the chat - both return the same
custom state directly (it's the same blob `getCustom()` reads server-side):

```ts
const chat = taskAgent.chat({
  state: { custom: { tasks: [], nextId: 1 }, messages: [], artifacts: [] },
});
const res = await chat.send('Add: buy groceries');

console.log(res.state);   // { tasks: [...], nextId: 2 }
console.log(chat.state);  // same - the latest custom state
```

On the client, use custom state to render application-specific UI alongside
the chat. For example, render a task list driven entirely by agent state:

```ts
// Client - render a task list from custom state
await chat.send(userMessage);
const tasks = chat.state?.tasks ?? [];
for (const task of tasks) {
  renderTaskItem(task.id, task.title, task.done);
}
```

For agents that mutate custom state mid-turn (like the
[custom research agent](#aidefinecustomagent)), the client receives live
updates: each streamed `chunk.custom` carries the full, post-patch custom
state with a fresh object reference, so you can use it for `===`-based change
detection:

```ts
for await (const chunk of turn.stream) {
  if (chunk.custom) {
    updateProgressUI(chunk.custom.status); // live status updates
  } else if (chunk.text) {
    appendToUI(chunk.text);
  }
}
```

### Artifacts

Tools can produce artifacts (generated files, code, etc.) via
`ai.currentSession().addArtifacts()`:

```ts
const generateFile = ai.defineTool(
  {
    name: 'generateFile',
    description: 'Add a generated code file to the workspace.',
    inputSchema: z.object({ name: z.string(), content: z.string() }),
    outputSchema: z.object({ status: z.string() }),
  },
  async (input) => {
    ai.currentSession().addArtifacts([{
      name: input.name,
      parts: [{ text: input.content }],
    }]);
    return { status: `Artifact ${input.name} added.` };
  }
);
```

Artifacts are available as `res.artifacts` and `chat.artifacts` on the
client, stream in as `chunk.artifact`, and are persisted in the session
state.

## Agents with interrupts

Agents work seamlessly with [interrupts](/docs/interrupts) for human-in-the-loop
workflows - approvals, confirmations, or any time you need user input
mid-flow. When a turn pauses on an interrupt, the paused tool requests are
surfaced on `res.interrupts` as `AgentInterrupt` objects with `respond()`
and `restart()` builders. Genkit supports two interrupt resumption patterns:

| | Respond | Restart |
|---|---|---|
| **When to use** | [`defineInterrupt`](/docs/interrupts#define-manual-response-interrupts) tools - you provide the answer directly | [Approval-gated tools](/docs/interrupts#tools-with-restartable-interrupts) - the tool re-executes with metadata |
| **Tool executes?** | No - you supply the output | Yes - re-runs with `resumed` metadata |
| **Builder** | `interrupt.respond(output)` | `interrupt.restart()` |
| **Resume with** | `chat.resume({ respond: [...] })` | `chat.resume({ restart: [...] })` |

See [Interrupts](/docs/interrupts) for full details on defining both kinds
of interrupt tools. The examples below show both patterns with an agent.

### Respond pattern

Use this for [`defineInterrupt`](/docs/interrupts#define-manual-response-interrupts)
tools - the tool never executes server-side; you provide the output directly.
This is useful for collecting user input (clarification, choices, feedback):

**Server:**

```ts
const userApproval = ai.defineInterrupt({
  name: 'userApproval',
  description: 'Ask the user for approval before a sensitive action.',
  inputSchema: z.object({
    action: z.string(),
    details: z.string(),
  }),
  outputSchema: z.object({
    approved: z.boolean(),
    feedback: z.string().optional(),
  }),
});

const bankingAgent = ai.defineAgent({
  name: 'bankingAgent',
  system:
    'You are a banking assistant. ALWAYS use the userApproval interrupt to ' +
    'confirm details before executing the transferMoney tool.',
  tools: [userApproval, transferMoney],
  store: new InMemorySessionStore(),
});
```

**Client - detect and resume:**

```ts
const agent = remoteAgent({ url: '/api/bankingAgent' });
const chat = agent.chat();

let res = await chat.send('Transfer $500 to my savings account');

// Check if the agent paused on the userApproval interrupt.
const approval = res.interrupts.find((i) => i.name === 'userApproval');

if (approval) {
  // Show approval.input in the UI, collect the user's decision, then resume.
  // `respond()` builds the toolResponse part for you - the tool never runs.
  res = await chat.resume({
    respond: [approval.respond({ approved: true, feedback: 'Looks good' })],
  });
  console.log(res.text); // model's confirmation
}
```

Use `chat.resumeStream({ respond: [...] })` for a streamed resumption.

### Restart pattern

Use this for approval-gated tools where the tool itself checks for
explicit user approval before executing a sensitive action. This is
**more secure** than the respond pattern because the `resumed` metadata
can only be set by the client - the LLM can never forge it, so the
model cannot bypass the approval gate through prompt injection or
jailbreaking.

**Server - define a tool with an approval gate:**

```ts
const transferMoney = ai.defineTool(
  {
    name: 'transferMoney',
    description: 'Transfer money to a specified account.',
    inputSchema: z.object({ amount: z.number(), toAccount: z.string() }),
    outputSchema: z.object({ success: z.boolean(), transactionId: z.string() }),
  },
  async (input, ctx) => {
    // ctx.metadata.resumed can only be set by the client, never by the LLM.
    const isApproved = ctx.metadata?.resumed?.toolApproved === true;

    if (!isApproved && input.amount > 100) {
      // Interrupt - pause and return this tool request to the client.
      ctx.interrupt({
        message: `Transfer $${input.amount} to ${input.toAccount}. Approve?`,
        amount: input.amount,
      });
    }

    // Only reached if approved or amount ≤ $100.
    return { success: true, transactionId: `txn-${Date.now()}` };
  }
);
```

**Client - detect and restart with approval:**

```ts
let res = await chat.send('Transfer $500 to savings');

const transfer = res.interrupts.find((i) => i.name === 'transferMoney');

if (transfer) {
  // Show the transfer details and ask the user to approve. If approved,
  // restart the tool, attaching approval metadata. `restart()` re-issues the
  // original tool request; spread it and add the trusted `resumed` metadata.
  res = await chat.resume({
    restart: [{
      ...transfer.restart(),
      metadata: { resumed: { toolApproved: true } },
    }],
  });
  console.log(res.text); // model's confirmation
}
```

The key difference: with **respond**, you supply the tool's output and
the tool never runs. With **restart**, the tool re-executes - but this
time `ctx.metadata.resumed.toolApproved` is `true`, so it passes the
approval gate. This ensures the approval check lives in server-side
code that the LLM cannot circumvent.

Note that an arbitrary amount of time can pass between the interrupt and
the resumption - seconds, hours, or even days. When the agent interrupts,
the HTTP request completes normally and the server process is idle. All
session state is persisted in the store. When the client sends the
resumption request, it's a completely new HTTP request that loads the
session and continues where it left off. (To resume after a full page
reload, restore the chat first with `agent.loadChat({ snapshotId })`.)

## Background execution

For long-running tasks (report generation, deep research), run the agent
in the background. The client submits a request, receives a handle with a
`snapshotId` immediately, and polls for the result.

### Server

Define the agent with a store (required for background execution) and
expose the state and abort endpoints:

```ts
const backgroundAgent = ai.defineAgent({
  name: 'reportAgent',
  system: 'You are a research analyst. Write comprehensive reports.',
  store: new InMemorySessionStore(), // Required for detach
});
```

```ts
app.post('/api/backgroundAgent', expressHandler(backgroundAgent));
app.post('/api/backgroundAgent/state', expressHandler(backgroundAgent.getSnapshotDataAction));
app.post('/api/backgroundAgent/abort', expressHandler(backgroundAgent.abortAgentAction));
```

### Client

**1. Submit with `chat.detach()`** - the server returns a `DetachedTask`
handle immediately:

```ts
const agent = remoteAgent({ url: '/api/backgroundAgent' });

const task = await agent.chat().detach('Write a report on AI trends');
console.log(task.snapshotId); // available right away
```

**2. Poll for completion** with `task.poll()`, which yields snapshots until
a terminal status, or `task.wait()`, which resolves with the final snapshot:

```ts
for await (const snapshot of task.poll({ intervalMs: 2000 })) {
  // snapshot.status: 'pending' | 'done' | 'failed' | 'aborted'
  if (snapshot.status === 'done') {
    const messages = snapshot.state.messages;
    const modelReply = messages.filter((m) => m.role === 'model').pop();
    displayReport(modelReply);
  }
}

// Or, simply await the terminal snapshot:
const finalSnapshot = await task.wait({ intervalMs: 2000 });
```

**3. Abort if needed:**

```ts
await task.abort();
```

## Multi-agent delegation

Use the `agents()` middleware from `@genkit-ai/middleware` to let an
orchestrator agent delegate tasks to specialized sub-agents. The setup
involves defining sub-agents, wiring them into an orchestrator with the
middleware, and exposing the orchestrator as an endpoint:

### Server

```ts
import { agents } from '@genkit-ai/middleware';

// Define specialized sub-agents. The `description` is used by the
// orchestrator to decide when to delegate.
const researcher = ai.defineAgent({
  name: 'researcher',
  description: 'A thorough research assistant that finds well-sourced answers.',
  system: 'You are a thorough research assistant.',
  tools: [getWebResults],
});

const coder = ai.defineAgent({
  name: 'coder',
  description: 'An expert programmer that writes clean, well-commented code.',
  system: 'You are an expert programmer. Write clean, well-commented code.',
});

// Orchestrator delegates to sub-agents via the `agents` middleware.
const orchestratorAgent = ai.defineAgent({
  name: 'orchestrator',
  system: `You are a project assistant. Analyze the user's request and
delegate to the appropriate sub-agent, then synthesize a final answer.`,
  use: [
    agents({
      agents: [
        'researcher', // description auto-discovered from the agent
        { name: 'coder', description: 'Writes and debugs code.' }, // override
      ],
      maxDelegations: 5,  // guard rail against runaway loops
      historyLength: 4,   // forward conversation context to sub-agents
    }),
  ],
});
```

The middleware injects a `call_agent` tool. When the orchestrator model
calls it, the middleware runs the sub-agent (via its `.run()` method) and
returns the response as the tool result. Interrupts from sub-agents
propagate up automatically.

### Client

The client consumes the orchestrator exactly like any other agent. The
`call_agent` tool calls appear in the stream, so you can show delegation
status in the UI:

```ts
const agent = remoteAgent({ url: '/api/orchestratorAgent' });
const chat = agent.chat();

const turn = chat.sendStream(
  'Research sorting algorithms then write a quicksort in TypeScript.'
);

for await (const chunk of turn.stream) {
  if (chunk.text) appendToUI(chunk.text);

  for (const part of chunk.raw.modelChunk?.content ?? []) {
    if (part.toolRequest?.name === 'call_agent') {
      showStatus(`Delegating to ${part.toolRequest.input.agent}…`);
    }
    if (part.toolResponse?.name === 'call_agent') {
      showStatus('Sub-agent responded');
    }
  }
}

const res = await turn.response;
```

## Error handling

When a turn fails, the chat throws an `AgentError`. It carries the failure
details plus the **last-good state**, so the session stays recoverable - the
user can keep chatting after an error:

```ts
import { AgentError } from 'genkit/beta/client';

try {
  const res = await chat.send(userMessage);
  // ...
} catch (err) {
  if (err instanceof AgentError) {
    console.log(err.status);     // e.g. 'INTERNAL', 'NOT_FOUND'
    console.log(err.message);
    console.log(err.snapshotId); // last-good snapshot (server-managed)
    console.log(err.state);      // last-good custom state (client-managed)
    // The chat is still usable - the next send() resumes from last-good state.
  } else {
    throw err;
  }
}
```

Because the last-good snapshot/state is preserved, you can surface the error
in the UI and let the user retry without losing the conversation. The same
`AgentError` is thrown by the in-process server agent and the HTTP
`remoteAgent()` client.
