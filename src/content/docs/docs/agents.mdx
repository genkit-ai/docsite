Build chatbots, approval flows, and background generators with Genkit
Agents. You define a prompt, tools, and state once, and Genkit handles
the loop - message history, tool calls, streaming, and persistence - so
you can focus on your product.

You start with a few lines of code and grow into tools, custom state,
streaming, persistence, and multi-agent systems only as you need them.
The same agent object behaves identically whether you call it in-process
or over HTTP, so your server and client code look the same.

Agents are built on prompts and the `generate()` API, so they have full
access to [middleware](/docs/middleware) - retries, fallbacks, tool
approval, and custom hooks. They replace the deprecated
[`ai.chat()`](/docs/chat) API.

All of the code samples in this guide can be found in the [agents sample](https://github.com/genkit-ai/genkit/tree/main/js/testapps/agents) on GitHub.

## Before you begin

Before reading this page, you should be familiar with [generating content](/docs/models),
[tool calling](/docs/tool-calling), and [prompts](/docs/dotprompt).

:::note[Beta import]
All agent APIs are imported from **`genkit/beta`** (server) and
**`genkit/beta/client`** (client), not `genkit`. This applies to
`genkit()`, session stores, `remoteAgent()`, and the shared agent types.
:::

## Your first agent

The simplest agent is just a name and a system prompt. Define it, start a chat,
and send a message:

```ts
import { genkit } from 'genkit/beta';
import { googleAI } from '@genkit-ai/google-genai';

const ai = genkit({
  plugins: [googleAI()],
  model: googleAI.model('gemini-flash-latest'),
});

const assistant = ai.defineAgent({
  name: 'assistant',
  system: 'You are a helpful assistant.',
});

async function main() {
  const chat = assistant.chat();
  const res = await chat.send('Hello! What can you do?');
  console.log(res.text);
}

main();
```

That's a complete, working agent. Everything else on this page is about adding
capabilities to this foundation as you need them.

:::note[Referencing models]
You can reference a model either with the plugin helper
(`googleAI.model('gemini-flash-latest')`) or with its string ID
(`'googleai/gemini-flash-latest'`). The two are interchangeable; this page
uses both.
:::

## Add tools and serve over HTTP

Most agents do two more things: call **tools** to take actions, and run as an
HTTP **endpoint** your app can reach. Here's the same kind of agent with a tool
added and served over HTTP - still a single file.

You'll need the `genkit`, `@genkit-ai/google-genai`, `@genkit-ai/express`, and
`express` packages, plus a Gemini API key set as `GEMINI_API_KEY` (you can get
one from [Google AI Studio](https://aistudio.google.com/apikey)).

```ts
import { genkit, z } from 'genkit/beta';
import { googleAI } from '@genkit-ai/google-genai';
import { expressHandler } from '@genkit-ai/express';
import express from 'express';

const ai = genkit({
  plugins: [googleAI()],
  model: googleAI.model('gemini-flash-latest'),
});

// A tool the agent can call.
const getWeather = ai.defineTool(
  {
    name: 'getWeather',
    description: 'Get the current weather for a location.',
    inputSchema: z.object({ location: z.string() }),
    outputSchema: z.object({ temperatureF: z.number(), conditions: z.string() }),
  },
  async ({ location }) => {
    // Replace with a real weather API call.
    return { temperatureF: 72, conditions: `sunny in ${location}` };
  }
);

// Define the agent.
const weatherAgent = ai.defineAgent({
  name: 'weatherAgent',
  system: 'You are a helpful weather assistant. Answer weather queries.',
  tools: [getWeather],
});

// Serve it over HTTP.
const app = express();
app.use(express.json());
app.post('/api/weatherAgent', expressHandler(weatherAgent));
app.listen(8080, () => console.log('Agent listening on http://localhost:8080'));
```

That's a working, tool-using agent served over HTTP - and you can test it in
the [developer UI](#testing-agents-in-the-developer-ui) without writing any
client code. From here, Genkit grows with what you need:

- **Persist conversations** so users can resume them later → [Session persistence](#session-persistence)
- **Stream responses** token by token to your UI → [Multi-turn streaming chat](#multi-turn-streaming-chat)
- **Track structured app state** like lists, counters, or records → [Custom state and artifacts](#custom-state-and-artifacts)
- **Pause for human approval** mid-turn → [Agents with interrupts](#agents-with-interrupts)
- **Run long tasks in the background** → [Background execution](#background-execution)
- **Coordinate multiple specialized agents** → [Multi-agent delegation](#multi-agent-delegation)

Each of these builds on the same agent you just wrote; add only what you need.

## Defining agents

`ai.defineAgent()` is how you'll define almost every agent - it's what you used
above. Give it a system prompt and, optionally, tools, and Genkit runs the
model and tool-calling loop for you.

Two variants exist for when you need more:

- **`ai.definePromptAgent()`** - same as `defineAgent`, but the prompt lives
  in a [Dotprompt](/docs/dotprompt) `.prompt` file instead of inline. Reach for
  it when prompts are authored or versioned separately from code.
- **`ai.defineCustomAgent()`** - full control over orchestration (multiple
  model calls per turn, custom progress streaming, direct session
  manipulation). Reach for it only when `defineAgent` isn't flexible enough.

### `ai.defineAgent()`

The primary way to create an agent. Define a model, system prompt, tools, and
optionally a **session store** - all in one place. A session store is the
persistence backend that saves a conversation's message history and state
between requests, so a session can be resumed later (see
[Session persistence](#session-persistence)):

```ts
import { FileSessionStore } from 'genkit/beta';

const weatherAgent = ai.defineAgent({
  name: 'weatherAgent',
  system: 'You are a helpful weather assistant. Answer weather queries.',
  tools: [getWeather],
  // Persist session state between requests (see "Session persistence" section)
  store: new FileSessionStore('./.snapshots'),
});
```

The `model` is optional - it defaults to the model configured in `genkit()`.
Pass `model` to override it per-agent.

**With typed custom state:**

Define your state shape with a Zod schema and pass it as `stateSchema`.
This gives you TypeScript types and runtime validation. Genkit also
generates JSON Schema from the Zod schema and uses it in the developer
UI, where you can view and edit custom state directly while testing
your agent:

```ts
const TaskStateSchema = z.object({
  tasks: z.array(z.object({
    id: z.number(),
    title: z.string(),
    done: z.boolean(),
  })),
  nextId: z.number(),
});

type TaskState = z.infer<typeof TaskStateSchema>;

const taskAgent = ai.defineAgent({
  name: 'taskAgent',
  stateSchema: TaskStateSchema,
  system: 'You are a task management assistant. Use the provided tools.',
  tools: [addTask, toggleTask, removeTask],
});
```

### `ai.definePromptAgent()`

Wire an agent from a [Dotprompt](/docs/dotprompt) `.prompt` file. This
separates the prompt template from agent wiring, which is useful when:

- Non-developer prompt authors edit templates independently
- You want versioned, reviewable prompt files in source control
- Content teams manage prompt copy without touching application code
- You need to A/B test different prompt variants

```handlebars
---
model: googleai/gemini-flash-latest
tools:
  - getAttractions
  - getFlightInfo
---

{{role "system"}}
You are a friendly trip planning assistant. Help users plan trips by
suggesting attractions and looking up flight information.

{{history}}
```

```ts
const tripPlannerAgent = ai.definePromptAgent({
  promptName: 'tripPlanner',
  store: new FileSessionStore('./.snapshots'),
});
```

For full control over orchestration - multiple model calls per turn, custom
progress streaming, and direct session manipulation - see
[Advanced: custom orchestration](#advanced-custom-orchestration) near the end
of this page. Most agents never need it; reach for `defineAgent` first.

## The unified agent API

Every agent - whether defined in-process with `ai.defineAgent()` or
accessed over HTTP from a client (via `remoteAgent()`, covered in
[Serving agents over HTTP](#serving-agents-over-http)) - exposes the same
surface. Once you have an agent, you start a `chat` and send turns:

```ts
// `weatherAgent` is the object returned by ai.defineAgent(...)
const chat = weatherAgent.chat();

// Streaming turn
const turn = chat.sendStream('What is the weather in Tokyo?');
for await (const chunk of turn.stream) {
  if (chunk.text) process.stdout.write(chunk.text); // streamed model text
}
const res = await turn.response;
console.log(res.text);        // final text
console.log(res.snapshotId);  // snapshot id (server-managed agents)
console.log(res.state);       // custom state

// Non-streaming follow-up turn - state is carried forward automatically.
const res2 = await chat.send('What about Paris?');
```

The chat tracks session continuity for you - no manual `snapshotId`/`state`
threading. The client-side `remoteAgent()` (below) returns the same object,
so client code looks identical (see [Serving agents over HTTP](#serving-agents-over-http)).

The core types:

| Type | What it is | Key members |
|---|---|---|
| `AgentAPI` | The agent itself | `chat()`, `loadChat()`, `getSnapshot()`, `abort()` |
| `AgentChat` | A stateful conversation | `send()`, `sendStream()`, `resume()`, `resumeStream()`, `detach()`, `abort()`, `snapshotId`, `state`, `messages`, `artifacts` |
| `AgentTurn` | One in-flight turn | `stream`, `response`, `abort()` |
| `AgentResponse` | A completed turn | `text`, `message`, `messages`, `toolRequests`, `interrupts`, `finishReason`, `snapshotId`, `state`, `artifacts`, `raw` |
| `AgentChunk` | A streamed chunk | `text`, `accumulatedText`, `reasoning`, `toolRequests`, `artifact`, `custom`, `raw` |
| `AgentInterrupt` | A paused tool request | `name`, `input`, `respond(output)`, `restart()` |
| `DetachedTask` | A background task handle | `snapshotId`, `poll()`, `wait()`, `abort()` |

`res.state` returns the **custom** state directly, while `res.raw.state`
holds the full `SessionState` (messages + custom + artifacts).

**Options for `chat()`.** Calling `chat()` with no arguments starts a fresh
conversation. You can also pass any of the following (they can be combined):

| Option | Type | Use it to |
|---|---|---|
| `state` | `{ custom, messages, artifacts }` | Seed initial state - custom state, prior messages, or artifacts (see [Custom state](#custom-state)) |
| `snapshotId` | `string` | Branch from an existing snapshot (server-managed; see [Session branching](#session-branching)) |
| `sessionId` | `string` | Continue a session from its latest snapshot (server-managed) |

To restore a full conversation (with history) rather than start one, use
`loadChat()` instead - see [Session persistence](#session-persistence).

## State management strategies

Every agent uses one of two strategies for maintaining session state across
turns. This choice affects how the client continues a conversation - so pick
a strategy before building your client.

| | Server-managed | Client-managed |
|---|---|---|
| **Setup** | Define the agent with a `store` | Omit `store` |
| **How a turn continues** | The chat tracks `snapshotId`; resume with a `snapshotId` or `sessionId` | The chat tracks and round-trips the full `state` blob |
| **Server stores state?** | Yes | No |
| **Supports branching?** | Yes | Manual (open multiple chats from the same `state`) |
| **Supports background execution?** | Yes | No |

In both cases the `chat` object carries continuity automatically; the
difference is *where* the state lives. See
[Session persistence](#session-persistence) for detailed examples of each
strategy, including session restoration, branching, and pruning.

## Serving agents over HTTP

Before your client app can talk to an agent, expose it as an HTTP endpoint.
Use `expressHandler` from `@genkit-ai/express`:

```ts
import { expressHandler } from '@genkit-ai/express';
import express from 'express';

const app = express();
app.use(express.json());

// Main agent endpoint - handles run, streaming, and multi-turn.
// This is the only endpoint required for every agent.
app.post('/api/weatherAgent', expressHandler(weatherAgent));

// Optional: snapshot endpoint - only for server-managed (persisted) agents.
// Enables clients to restore/inspect a session via loadChat()/getSnapshot().
app.post('/api/weatherAgent/state', expressHandler(weatherAgent.getSnapshotDataAction));

// Optional: abort endpoint - only needed for background (detached) execution,
// so clients can cancel an in-flight background task.
app.post('/api/weatherAgent/abort', expressHandler(weatherAgent.abortAgentAction));

app.listen(8080);
```

Every agent exposes one **primary endpoint** - that's all you need for
streaming chat and multi-turn conversations. The other two endpoints are
optional and only matter in specific cases:

- **`getSnapshotDataAction`** (the `/state` endpoint) - expose this only for
  **server-managed agents** (those defined with a `store`). It lets clients
  restore or inspect a persisted session via `loadChat()` / `getSnapshot()`.
  Client-managed (store-less) agents don't need it - the client already holds
  the full state.
- **`abortAgentAction`** (the `/abort` endpoint) - expose this only when you
  use [background execution](#background-execution), so clients can cancel an
  in-flight detached task. Regular foreground turns are aborted client-side
  (see `AgentTurn.abort()`) and don't require this endpoint.

Both `getSnapshotDataAction` and `abortAgentAction` are **properties on the
agent object** returned by `ai.defineAgent()` (server-side only) - they're
part of the agent's serving surface, not the shared `AgentAPI` interface
that mirrors across server and client.

On the **client side**, create a typed client with `remoteAgent()` from
`genkit/beta/client`:

```ts
import { remoteAgent } from 'genkit/beta/client';

const agent = remoteAgent({ url: '/api/weatherAgent' });

const chat = agent.chat();
const res = await chat.send('Weather in Tokyo?');
console.log(res.text);
```

`remoteAgent()` defaults the state and abort URLs to `${url}/state` and
`${url}/abort`. Override them (or add request headers) if your routes differ:

```ts
const agent = remoteAgent({
  url: '/api/weatherAgent',
  stateUrl: '/api/weatherAgent/state',   // optional
  abortUrl: '/api/weatherAgent/abort',   // optional
  headers: { Authorization: `Bearer ${token}` }, // static or async function
});
```

Because `remoteAgent()` returns the same `AgentAPI` as the in-process agent,
**your client code is the same code you'd write on the server** - just
pointed at a URL. The rest of this page shows both sides of each pattern.

## Multi-turn streaming chat

The most common pattern: a streaming chat where the client sends messages,
receives streamed tokens and tool-call events, and maintains session
continuity across turns.

### Server

Use `chat.sendStream()` and iterate the stream, then await the response:

```ts
const chat = weatherAgent.chat();

const turn = chat.sendStream('Weather in London?');
for await (const chunk of turn.stream) {
  if (chunk.text) process.stdout.write(chunk.text);
}
const res = await turn.response;

console.log(res.message);    // Model's response message
console.log(res.state);      // Custom state
console.log(res.snapshotId); // Snapshot ID (if using a store)

// Continue the conversation - the same chat carries state forward.
const res2 = await chat.send('What about Paris?');
```

### Client

The client code mirrors the server. Use `chat.sendStream()`. Each chunk
exposes streamed text via `chunk.text`, and you can read tool calls and
responses off the raw model chunk:

```ts
import { remoteAgent } from 'genkit/beta/client';

const agent = remoteAgent({ url: '/api/weatherAgent' });
const chat = agent.chat();

const turn = chat.sendStream(userMessage);

for await (const chunk of turn.stream) {
  if (chunk.text) appendToUI(chunk.text); // streamed model text

  // Tool calls and responses arrive on the raw model chunk.
  for (const part of chunk.raw.modelChunk?.content ?? []) {
    if (part.toolRequest) showToolCall(part.toolRequest);
    if (part.toolResponse) showToolResult(part.toolResponse);
  }
}

const res = await turn.response;
// chat.snapshotId / chat.state are updated automatically for the next turn.
```

`chunk.raw` is the underlying transport chunk - everything Genkit received
for that step before it was normalized into the convenience fields
(`chunk.text`, `chunk.custom`, etc.). Its `modelChunk` holds the raw model
response, including `content` parts like `toolRequest` and `toolResponse`,
which is why the loop above reads tool calls from `chunk.raw.modelChunk`.

`chunk.accumulatedText` gives you the running text so far, which is handy
for rendering a growing message bubble.

For non-streaming requests, use `chat.send()`:

```ts
const res = await chat.send('Weather in London?');
console.log(res.text);
```

## Session persistence

This section covers each [state management strategy](#state-management-strategies)
in detail - stores, session restoration, client-managed state, branching, and
pruning.

### Server-side stores

Use a store to persist session state across requests. Genkit provides
`FileSessionStore` and `InMemorySessionStore`:

```ts
import { FileSessionStore, InMemorySessionStore } from 'genkit/beta';

// File-based persistence
const agent = ai.defineAgent({
  name: 'myAgent',
  // ...
  store: new FileSessionStore('./.snapshots'),
});

// In-memory (for development/testing)
const agent = ai.defineAgent({
  name: 'myAgent',
  // ...
  store: new InMemorySessionStore(),
});
```

With a store, each turn produces a `snapshotId` (available as
`chat.snapshotId` and `res.snapshotId`) that you can use to restore the
session later - for example, when a user bookmarks a chat URL or returns to
a page.

**Restore a chat with `loadChat()`:**

`loadChat()` reads a snapshot and returns a ready-to-use chat with history
restored. Accept either a `snapshotId` (an exact snapshot) or a `sessionId`
(the session's latest snapshot):

```ts
// Client - restore from a snapshotId (e.g., from the URL)
const chat = await agent.loadChat({ snapshotId });

// History is restored - rebuild the UI from chat.messages
for (const msg of chat.messages) {
  for (const part of msg.content) {
    if (part.text) renderMessage(msg.role, part.text);
  }
}

// Continue the conversation right where it left off.
const res = await chat.send('And what about tomorrow?');
```

**Read a snapshot without starting a chat:**

When you only need the data (not a live chat), use `getSnapshot()`:

```ts
const snapshot = await agent.getSnapshot(snapshotId);
// snapshot.state.messages  - full message history
// snapshot.state.custom    - custom state
// snapshot.state.artifacts - artifacts
// snapshot.status          - 'pending' | 'done' | 'failed' | 'aborted'
```

This lets you store the `snapshotId` in the URL, a cookie, or a database,
and fully restore the conversation later. The same `loadChat()` /
`getSnapshot()` methods work in-process on the server and over HTTP via
`remoteAgent()`.

### Client-managed state

If you don't provide a `store`, the agent is stateless on the server and the
client owns the session `state`. The `chat` object still tracks state for you
and round-trips it on every turn - the only difference is that there's no
`snapshotId`:

```ts
// Server - no store
const clientAgent = ai.defineAgent({
  name: 'clientAgent',
  system: 'You are a helpful assistant.',
  tools: [getWeather],
  // No `store` property
});
```

```ts
// Client - the chat round-trips the state blob automatically.
const agent = remoteAgent({ url: '/api/clientAgent' });
const chat = agent.chat();

await chat.send('Weather in Tokyo?');
await chat.send('What about Paris?'); // state carried forward automatically

// The full client-owned state blob:
console.log(chat.state);          // custom state
// res.raw.state holds the entire SessionState (messages + custom + artifacts)
```

The trade-off: no server storage needed, but the client holds (and the
transport sends) the full message history on every request. This works well
for short conversations or when you want a fully stateless server.

To seed initial custom state, pass it to `chat()`:

```ts
const chat = agent.chat({
  state: { custom: { tasks: [], nextId: 1 }, messages: [], artifacts: [] },
});
```

### Session branching

With a store, you can branch conversations from any snapshot point - creating
parallel conversation threads from the same starting point. Open multiple
chats from the same `snapshotId`:

```ts
const root = agent.chat();
const res1 = await root.send('Hello!');
const branchPoint = res1.snapshotId;

// Branch A
const branchA = agent.chat({ snapshotId: branchPoint });
await branchA.send('My name is Alice.');

// Branch B - forks from the SAME snapshot
const branchB = agent.chat({ snapshotId: branchPoint });
await branchB.send('My name is Bob.');
```

A `snapshotId` is an immutable checkpoint - like a git commit, the original
doesn't change when you branch from it. Each branch creates new, independent
snapshots. This works identically with `remoteAgent()` on the client, which
is how you can, for example, generate two response variants in parallel and
let the user pick one:

```ts
// Client - two parallel turns from the same branch point
const makeChat = () => agent.chat({ snapshotId });
const [a, b] = await Promise.all([
  makeChat().send(text),
  makeChat().send(text),
]);
// a.snapshotId !== b.snapshotId - pick one to continue from.
```

### Snapshot chain pruning

Control how many snapshots are retained in the chain with
`maxPersistedChainLength`:

```ts
const store = new FileSessionStore('./.snapshots', {
  maxPersistedChainLength: 3, // Keep only the last 3 snapshots
});
```

## Custom state and artifacts

### Custom state

Agents don't just produce text - they produce **structured app state**
that drives your UI. Custom state is useful when your application needs a
reliable source of truth that the model can't maintain in chat history
alone - exact IDs, counters, structured records, or any data where
precision matters. Tools can read and mutate typed custom state via
`ai.currentSession()` (reusing the `TaskState` schema from
[`ai.defineAgent()`](#aidefineagent) above).

To **mutate** state, call `session.updateCustom()` with a function that
receives the current state and returns the new state:

```ts
// Types inferred from the TaskStateSchema defined earlier.
type TaskState = z.infer<typeof TaskStateSchema>;
type TaskItem = TaskState['tasks'][number];

const addTask = ai.defineTool(
  {
    name: 'addTask',
    description: 'Add a new task to the task list.',
    inputSchema: z.object({ title: z.string() }),
    outputSchema: z.object({ id: z.number(), title: z.string(), done: z.boolean() }),
  },
  async (input) => {
    const session = ai.currentSession<TaskState>();
    let newTask!: TaskItem;
    session.updateCustom((state) => {
      // Normalize each field. On a fresh session custom state is an empty
      // object `{}` (see note below), so `state.tasks` may be undefined -
      // default per-field rather than relying on `state || defaults`.
      const tasks = state?.tasks ?? [];
      const nextId = state?.nextId ?? 1;
      newTask = { id: nextId, title: input.title, done: false };
      return { tasks: [...tasks, newTask], nextId: nextId + 1 };
    });
    return newTask;
  }
);
```

To **read** state without mutating it - for example in a "list tasks" tool -
call `session.getCustom()`, which returns the current custom state (or
`undefined` if nothing has been set yet):

```ts
const listTasks = ai.defineTool(
  {
    name: 'listTasks',
    description: 'List all tasks.',
    inputSchema: z.object({}),
    outputSchema: z.object({
      tasks: z.array(
        z.object({ id: z.number(), title: z.string(), done: z.boolean() })
      ),
    }),
  },
  async () => {
    const session = ai.currentSession<TaskState>();
    const state = session.getCustom();
    return { tasks: state?.tasks ?? [] };
  }
);
```

Access custom state from the response or the chat - both return the same
custom state directly (it's the same blob `getCustom()` reads server-side):

```ts
const chat = taskAgent.chat({
  state: { custom: { tasks: [], nextId: 1 }, messages: [], artifacts: [] },
});
const res = await chat.send('Add: buy groceries');

console.log(res.state);   // { tasks: [...], nextId: 2 }
console.log(chat.state);  // same - the latest custom state
```

On the client, use custom state to render application-specific UI alongside
the chat. For example, render a task list driven entirely by agent state:

```ts
// Client - render a task list from custom state
await chat.send(userMessage);
const tasks = chat.state?.tasks ?? [];
for (const task of tasks) {
  renderTaskItem(task.id, task.title, task.done);
}
```

For agents that mutate custom state mid-turn (like the
[custom research agent](#advanced-custom-orchestration)), the client receives live
updates: each streamed `chunk.custom` carries the full, post-patch custom
state with a fresh object reference, so you can use it for `===`-based change
detection:

```ts
for await (const chunk of turn.stream) {
  if (chunk.custom) {
    updateProgressUI(chunk.custom.status); // live status updates
  } else if (chunk.text) {
    appendToUI(chunk.text);
  }
}
```

### Artifacts

Tools can produce artifacts (generated files, code, etc.) via
`ai.currentSession().addArtifacts()`:

```ts
const generateFile = ai.defineTool(
  {
    name: 'generateFile',
    description: 'Add a generated code file to the workspace.',
    inputSchema: z.object({ name: z.string(), content: z.string() }),
    outputSchema: z.object({ status: z.string() }),
  },
  async (input) => {
    ai.currentSession().addArtifacts([{
      name: input.name,
      parts: [{ text: input.content }],
    }]);
    return { status: `Artifact ${input.name} added.` };
  }
);
```

Artifacts are available as `res.artifacts` and `chat.artifacts` on the
client, stream in as `chunk.artifact`, and are persisted in the session
state.

## Agents with interrupts

Agents work seamlessly with [interrupts](/docs/interrupts) for human-in-the-loop
workflows - approvals, confirmations, or any time you need user input
mid-flow. When a turn pauses on an interrupt, the paused tool requests are
surfaced on `res.interrupts` as `AgentInterrupt` objects with `respond()`
and `restart()` builders. Genkit supports two interrupt resumption patterns:

| | Respond | Restart |
|---|---|---|
| **When to use** | [`defineInterrupt`](/docs/interrupts#define-manual-response-interrupts) tools - you provide the answer directly | [Approval-gated tools](/docs/interrupts#tools-with-restartable-interrupts) - the tool re-executes with metadata |
| **Tool executes?** | No - you supply the output | Yes - re-runs with `resumed` metadata |
| **Builder** | `interrupt.respond(output)` | `interrupt.restart()` |
| **Resume with** | `chat.resume({ respond: [...] })` | `chat.resume({ restart: [...] })` |

See [Interrupts](/docs/interrupts) for full details on defining both kinds
of interrupt tools. The examples below show both patterns with an agent.

### Respond pattern

Use this for [`defineInterrupt`](/docs/interrupts#define-manual-response-interrupts)
tools - the tool never executes server-side; you provide the output directly.
This is useful for collecting user input (clarification, choices, feedback):

**Server:**

```ts
const userApproval = ai.defineInterrupt({
  name: 'userApproval',
  description: 'Ask the user for approval before a sensitive action.',
  inputSchema: z.object({
    action: z.string(),
    details: z.string(),
  }),
  outputSchema: z.object({
    approved: z.boolean(),
    feedback: z.string().optional(),
  }),
});

const bankingAgent = ai.defineAgent({
  name: 'bankingAgent',
  system:
    'You are a banking assistant. ALWAYS use the userApproval interrupt to ' +
    'confirm details before executing the transferMoney tool.',
  tools: [userApproval, transferMoney],
  store: new InMemorySessionStore(),
});
```

**Client - detect and resume:**

```ts
const agent = remoteAgent({ url: '/api/bankingAgent' });
const chat = agent.chat();

let res = await chat.send('Transfer $500 to my savings account');

// Check if the agent paused on the userApproval interrupt.
const approval = res.interrupts.find((i) => i.name === 'userApproval');

if (approval) {
  // Show approval.input in the UI, collect the user's decision, then resume.
  // `respond()` builds the toolResponse part for you - the tool never runs.
  res = await chat.resume({
    respond: [approval.respond({ approved: true, feedback: 'Looks good' })],
  });
  console.log(res.text); // model's confirmation
}
```

Use `chat.resumeStream({ respond: [...] })` for a streamed resumption.

### Restart pattern

Use this for approval-gated tools where the tool itself checks for
explicit user approval before executing a sensitive action. This is
**more secure** than the respond pattern because the `resumed` metadata
can only be set by the client - the LLM can never forge it, so the
model cannot bypass the approval gate through prompt injection or
jailbreaking.

**Server - define a tool with an approval gate:**

```ts
const transferMoney = ai.defineTool(
  {
    name: 'transferMoney',
    description: 'Transfer money to a specified account.',
    inputSchema: z.object({ amount: z.number(), toAccount: z.string() }),
    outputSchema: z.object({ success: z.boolean(), transactionId: z.string() }),
  },
  async (input, ctx) => {
    // ctx.metadata.resumed can only be set by the client, never by the LLM.
    const isApproved = ctx.metadata?.resumed?.toolApproved === true;

    if (!isApproved && input.amount > 100) {
      // Interrupt - pause and return this tool request to the client.
      ctx.interrupt({
        message: `Transfer $${input.amount} to ${input.toAccount}. Approve?`,
        amount: input.amount,
      });
    }

    // Only reached if approved or amount ≤ $100.
    return { success: true, transactionId: `txn-${Date.now()}` };
  }
);
```

**Client - detect and restart with approval:**

```ts
let res = await chat.send('Transfer $500 to savings');

const transfer = res.interrupts.find((i) => i.name === 'transferMoney');

if (transfer) {
  // Show the transfer details and ask the user to approve. If approved,
  // restart the tool, attaching approval metadata. `restart()` re-issues the
  // original tool request; spread it and add the trusted `resumed` metadata.
  res = await chat.resume({
    restart: [{
      ...transfer.restart(),
      metadata: { resumed: { toolApproved: true } },
    }],
  });
  console.log(res.text); // model's confirmation
}
```

The key difference: with **respond**, you supply the tool's output and
the tool never runs. With **restart**, the tool re-executes - but this
time `ctx.metadata.resumed.toolApproved` is `true`, so it passes the
approval gate. This ensures the approval check lives in server-side
code that the LLM cannot circumvent.

Note that an arbitrary amount of time can pass between the interrupt and
the resumption - seconds, hours, or even days. When the agent interrupts,
the HTTP request completes normally and the server process is idle. All
session state is persisted in the store. When the client sends the
resumption request, it's a completely new HTTP request that loads the
session and continues where it left off. (To resume after a full page
reload, restore the chat first with `agent.loadChat({ snapshotId })`.)

## Background execution

For long-running tasks (report generation, deep research), run the agent
in the background. The client submits a request, receives a handle with a
`snapshotId` immediately, and polls for the result.

### Server

Define the agent with a store (required for background execution) and
expose the state and abort endpoints:

```ts
const backgroundAgent = ai.defineAgent({
  name: 'reportAgent',
  system: 'You are a research analyst. Write comprehensive reports.',
  store: new InMemorySessionStore(), // Required for detach
});
```

```ts
app.post('/api/backgroundAgent', expressHandler(backgroundAgent));
app.post('/api/backgroundAgent/state', expressHandler(backgroundAgent.getSnapshotDataAction));
app.post('/api/backgroundAgent/abort', expressHandler(backgroundAgent.abortAgentAction));
```

### Client

**1. Submit with `chat.detach()`** - the server returns a `DetachedTask`
handle immediately:

```ts
const agent = remoteAgent({ url: '/api/backgroundAgent' });

const task = await agent.chat().detach('Write a report on AI trends');
console.log(task.snapshotId); // available right away
```

**2. Poll for completion** with `task.poll()`, which yields snapshots until
a terminal status, or `task.wait()`, which resolves with the final snapshot:

```ts
for await (const snapshot of task.poll({ intervalMs: 2000 })) {
  // snapshot.status: 'pending' | 'done' | 'failed' | 'aborted'
  if (snapshot.status === 'done') {
    const messages = snapshot.state.messages;
    const modelReply = messages.filter((m) => m.role === 'model').pop();
    displayReport(modelReply);
  }
}

// Or, simply await the terminal snapshot:
const finalSnapshot = await task.wait({ intervalMs: 2000 });
```

**3. Abort if needed:**

```ts
await task.abort();
```

## Multi-agent delegation

Use the `agents()` middleware from `@genkit-ai/middleware` to let an
orchestrator agent delegate tasks to specialized sub-agents. The setup
involves defining sub-agents, wiring them into an orchestrator with the
middleware, and exposing the orchestrator as an endpoint:

### Server

```ts
import { agents } from '@genkit-ai/middleware';

// Define specialized sub-agents. The `description` is used by the
// orchestrator to decide when to delegate.
const researcher = ai.defineAgent({
  name: 'researcher',
  description: 'A thorough research assistant that finds well-sourced answers.',
  system: 'You are a thorough research assistant.',
  tools: [getWebResults],
});

const coder = ai.defineAgent({
  name: 'coder',
  description: 'An expert programmer that writes clean, well-commented code.',
  system: 'You are an expert programmer. Write clean, well-commented code.',
});

// Orchestrator delegates to sub-agents via the `agents` middleware.
const orchestratorAgent = ai.defineAgent({
  name: 'orchestrator',
  system: `You are a project assistant. Analyze the user's request and
delegate to the appropriate sub-agent, then synthesize a final answer.`,
  use: [
    agents({
      agents: [
        'researcher', // description auto-discovered from the agent
        { name: 'coder', description: 'Writes and debugs code.' }, // override
      ],
      maxDelegations: 5,  // guard rail against runaway loops
      historyLength: 4,   // forward conversation context to sub-agents
    }),
  ],
});
```

The middleware injects a `call_agent` tool. When the orchestrator model
calls it, the middleware runs the sub-agent (via its `.run()` method) and
returns the response as the tool result. Interrupts from sub-agents
propagate up automatically.

:::note[`.run()` vs `.chat()`]
`.run()` is the agent's low-level execution entry point - it runs a single
turn and is used internally by the runtime (for example, by the `agents()`
middleware to invoke a sub-agent). In your own application code, use
`.chat()` instead: it manages session continuity, state, and streaming for
you. You generally never need to call `.run()` directly.
:::

### Client

The client consumes the orchestrator exactly like any other agent. The
`call_agent` tool calls appear in the stream, so you can show delegation
status in the UI:

```ts
const agent = remoteAgent({ url: '/api/orchestratorAgent' });
const chat = agent.chat();

const turn = chat.sendStream(
  'Research sorting algorithms then write a quicksort in TypeScript.'
);

for await (const chunk of turn.stream) {
  if (chunk.text) appendToUI(chunk.text);

  for (const part of chunk.raw.modelChunk?.content ?? []) {
    if (part.toolRequest?.name === 'call_agent') {
      showStatus(`Delegating to ${part.toolRequest.input.agent}…`);
    }
    if (part.toolResponse?.name === 'call_agent') {
      showStatus('Sub-agent responded');
    }
  }
}

const res = await turn.response;
```

## Advanced: custom orchestration

Most agents are well served by `ai.defineAgent()`, which runs one model call
with an automatic tool-calling loop. When you need full control over what
happens within a turn, use `ai.defineCustomAgent()`. It lets you:

- **Chain multiple model calls** with custom logic between them - for
  example, decompose a question, research each part, then synthesize
- **Stream custom status to the client** by mutating typed custom state -
  the runtime auto-emits the change so the UI can show progress like
  "Researching step 2 of 3…"
- **Manipulate the session directly** with `addMessages` and `updateCustom`,
  giving you control over exactly what gets persisted
- **Use different models for different steps** - a fast model for
  decomposition, a capable model for synthesis

The following example shows a research agent that decomposes a question,
researches each part, and streams status updates throughout. Notice that
status is streamed by mutating custom state with
`session.updateCustom()` - the runtime auto-emits a `customPatch` chunk
that the client receives as `chunk.custom`:

```ts
interface ResearchState {
  status?: string;
  subQuestions: string[];
  subAnswers: Array<{ question: string; answer: string }>;
}

const researchAgent = ai.defineCustomAgent(
  { name: 'researchAgent' },
  async (sess, { sendChunk }) => {
    let lastMessage;
    const session = ai.currentSession<ResearchState>();

    // sess.run() handles loading and saving session state around your logic.
    await sess.run(async (input) => {
      // Extract the user's question from the latest message.
      const userText =
        input.messages?.[input.messages.length - 1]?.content[0]?.text || '';

      // Step 1: Use a fast model to decompose the question into sub-questions.
      // Mutating custom state auto-emits a `customPatch` chunk to the client.
      session.updateCustom((s) => ({ ...s!, status: 'Decomposing question…' }));
      const decompose = await ai.generate({
        model: 'googleai/gemini-flash-lite-latest',
        prompt: `Break this into 2-3 sub-questions as a JSON array: "${userText}"`,
        output: { format: 'json', schema: z.array(z.string()).min(2).max(3) },
      });
      const subQuestions = decompose.output ?? [userText];

      // Step 2: Research each sub-question with the default model.
      const subAnswers = [];
      for (let i = 0; i < subQuestions.length; i++) {
        session.updateCustom((s) => ({
          ...s!,
          status: `Researching (${i + 1}/${subQuestions.length})…`,
        }));
        const research = await ai.generate({
          prompt: `Answer concisely: ${subQuestions[i]}`,
        });
        subAnswers.push({ question: subQuestions[i], answer: research.text });
      }

      // Step 3: Synthesize findings - stream the final response token by token.
      session.updateCustom((s) => ({ ...s!, status: 'Synthesizing…' }));
      const synthesisStream = ai.generateStream({
        prompt: `Synthesize these findings into a cohesive answer:\n${
          subAnswers.map((sa) => `Q: ${sa.question}\nA: ${sa.answer}`).join('\n\n')
        }`,
      });

      // Forward each model chunk to the client as it arrives.
      for await (const chunk of synthesisStream.stream) {
        sendChunk({ modelChunk: chunk });
      }

      // Persist the final model message in the session history.
      lastMessage = (await synthesisStream.response).message;
      sess.addMessages([lastMessage]);
    });

    return {
      message: lastMessage || {
        role: 'model',
        content: [{ text: 'Research complete.' }],
      },
    };
  }
);
```

A custom agent's `sendChunk` only accepts the agent stream-chunk shape
(`modelChunk`, `customPatch`, `artifact`, `turnEnd`). To stream
human-readable progress, mutate custom state with `session.updateCustom()`
as shown above rather than sending an ad-hoc status field.

## Error handling

When a turn fails, the chat throws an `AgentError`. It carries the failure
details plus the **last-good state**, so the session stays recoverable - the
user can keep chatting after an error:

```ts
import { AgentError } from 'genkit/beta/client';

try {
  const res = await chat.send(userMessage);
  // ...
} catch (err) {
  if (err instanceof AgentError) {
    console.log(err.status);     // e.g. 'INTERNAL', 'NOT_FOUND'
    console.log(err.message);
    console.log(err.snapshotId); // last-good snapshot (server-managed)
    console.log(err.state);      // last-good custom state (client-managed)
    // The chat is still usable - the next send() resumes from last-good state.
  } else {
    throw err;
  }
}
```

Because the last-good snapshot/state is preserved, you can surface the error
in the UI and let the user retry without losing the conversation. The same
`AgentError` is thrown by the in-process server agent and the HTTP
`remoteAgent()` client.

### When a tool throws

A tool that throws an unhandled error aborts the turn the same way any other
failure does: it propagates out as an `AgentError`, and the session is left at
its last-good state (handle it with the `try`/`catch` above). The error does
**not** get fed back to the model, so the model won't see it or attempt to
recover on its own.

If instead you want the model to react to a failure - for example, retry with
different arguments or tell the user an item wasn't found - don't throw.
Return a normal result that describes the outcome so the model can act on it:

```ts
const checkInHabit = ai.defineTool(
  {
    name: 'checkInHabit',
    description: 'Mark a habit as completed for today.',
    inputSchema: z.object({ id: z.number() }),
    // Model the failure case in the output schema instead of throwing.
    outputSchema: z.object({
      ok: z.boolean(),
      error: z.string().optional(),
    }),
  },
  async ({ id }) => {
    const habit = findHabit(id);
    if (!habit) {
      // The model sees this and can tell the user the id wasn't found.
      return { ok: false, error: `No habit with id ${id}.` };
    }
    habit.checkedInToday = true;
    return { ok: true };
  }
);
```

Reserve `throw` for genuinely unrecoverable conditions where you want the
whole turn to fail; model expected, recoverable outcomes as tool output.

## Your first agent

The simplest agent is just a name and a system prompt. Define it, then run a
turn:

```go
package main

import (
	"context"
	"fmt"
	"log"

	"github.com/firebase/genkit/go/ai"
	aix "github.com/firebase/genkit/go/ai/exp"
	"github.com/firebase/genkit/go/genkit"
	"github.com/firebase/genkit/go/plugins/googlegenai"
)

func main() {
	ctx := context.Background()

	g := genkit.Init(ctx,
		genkit.WithPlugins(&googlegenai.GoogleAI{}),
		genkit.WithDefaultModel("googleai/gemini-flash-latest"),
	)

	assistant := genkit.DefineAgent[any](g, "assistant",
		aix.InlinePrompt{
			ai.WithSystem("You are a helpful assistant."),
		},
	)

	out, err := assistant.RunText(ctx, "Hello! What can you do?")
	if err != nil {
		log.Fatal(err)
	}
	fmt.Println(out.Message.Text())
}
```

That's a complete, working agent. Everything else on this page is about adding
capabilities to this foundation as you need them.

:::note[The `[any]` type parameter]
`DefineAgent` is generic over the agent's **custom state** type. With no
custom state, that type is `any`, and Go cannot infer it from zero arguments,
so the first example spells it `[any]` explicitly. Once you add a typed option
(such as a `store`) or use a state struct, Go infers it and you can drop the
parameter. See [Custom state](#custom-state-and-artifacts).
:::

## Add tools and serve over HTTP

Most agents do two more things: call **tools** to take actions, and run as an
HTTP **endpoint** your app can reach. Here's the same kind of agent with a tool
added and served over HTTP, still a single file.

```go
package main

import (
	"context"
	"log"
	"net/http"

	"github.com/firebase/genkit/go/ai"
	aix "github.com/firebase/genkit/go/ai/exp"
	"github.com/firebase/genkit/go/genkit"
	genkitx "github.com/firebase/genkit/go/genkit/exp"
	"github.com/firebase/genkit/go/plugins/googlegenai"
	"github.com/firebase/genkit/go/plugins/server"
)

type WeatherInput struct {
	Location string `json:"location"`
}

type WeatherOutput struct {
	TemperatureF float64 `json:"temperatureF"`
	Conditions   string  `json:"conditions"`
}

func main() {
	ctx := context.Background()

	g := genkit.Init(ctx,
		genkit.WithPlugins(&googlegenai.GoogleAI{}),
		genkit.WithDefaultModel("googleai/gemini-flash-latest"),
	)

	// A tool the agent can call.
	getWeather := genkit.DefineTool(g, "getWeather",
		"Get the current weather for a location.",
		func(ctx *ai.ToolContext, in WeatherInput) (WeatherOutput, error) {
			// Replace with a real weather API call.
			return WeatherOutput{TemperatureF: 72, Conditions: "sunny in " + in.Location}, nil
		},
	)

	// Define the agent, attaching the tool to its inline prompt.
	weatherAgent := genkit.DefineAgent[any](g, "weatherAgent",
		aix.InlinePrompt{
			ai.WithSystem("You are a helpful weather assistant. Answer weather queries."),
			ai.WithTools(getWeather),
		},
	)

	// Serve it over HTTP: range over the agent's default route layout and
	// mount each route onto a standard net/http mux.
	mux := http.NewServeMux()
	for _, route := range genkitx.AgentRoutes(weatherAgent) {
		mux.HandleFunc(route.Pattern(), route.Handler())
	}
	log.Fatal(server.Start(ctx, "127.0.0.1:8080", mux))
}
```

That's a working, tool-using agent served over HTTP, and you can test it in
the [developer UI](#testing-agents-in-the-developer-ui) without writing any
client code. From here, Genkit grows with what you need:

- **Persist conversations** so users can resume them later → [Session persistence](#session-persistence)
- **Stream responses** token by token to your UI → [Multi-turn streaming chat](#multi-turn-streaming-chat)
- **Track structured app state** like lists, counters, or records → [Custom state and artifacts](#custom-state-and-artifacts)
- **Pause for human approval** mid-turn → [Agents with interrupts](#agents-with-interrupts)
- **Run long tasks in the background** → [Background execution](#background-execution)
- **Coordinate multiple specialized agents** → [Multi-agent delegation](#multi-agent-delegation)

Each of these builds on the same agent you just wrote; add only what you need.

## Defining agents

`genkit.DefineAgent` is how you'll define almost every agent - it's what you
used above. Give it an inline prompt (`aix.InlinePrompt`) with a system prompt
and, optionally, tools, and Genkit runs the model and tool-calling loop for you.

Two variants exist for when you need more:

- **`genkit.DefinePromptAgent`** - same as `DefineAgent`, but the prompt lives
  in a [Dotprompt](/docs/dotprompt) `.prompt` file instead of inline. Reach for
  it when prompts are authored or versioned separately from code.
- **`genkit.DefineCustomAgent`** - full control over orchestration (multiple
  model calls per turn, custom progress streaming, direct session
  manipulation). Reach for it only when `DefineAgent` isn't flexible enough.

### `genkit.DefineAgent`

The primary way to create an agent. Its third argument is an `aix.InlinePrompt`
- a list of prompt options (model, system prompt, tools) that defines the prompt
inline. The `model` is optional; it defaults to the model configured with
`genkit.WithDefaultModel`. Add a `store` to persist the conversation (see
[Session persistence](#session-persistence)) and a description for the Dev UI:

```go
store, err := localstore.NewFileSessionStore[any]("./.genkit/snapshots/weatherAgent")
if err != nil {
	log.Fatal(err)
}

weatherAgent := genkit.DefineAgent(g, "weatherAgent",
	aix.InlinePrompt{
		ai.WithSystem("You are a helpful weather assistant. Answer weather queries."),
		ai.WithTools(getWeather),
	},
	// Persist session state between requests (see "Session persistence").
	aix.WithSessionStore(store),
	aix.WithDescription[any]("Answers weather queries."),
)
```

Note that once a typed option such as `WithSessionStore(store)` is present, Go
infers the state type from it, so the `[any]` on `DefineAgent` is no longer
needed.

### `genkit.DefinePromptAgent`

Wire an agent from a [Dotprompt](/docs/dotprompt) `.prompt` file instead of an
inline prompt. This separates the prompt template from agent wiring, which is
useful when prompt authors are not the people writing the Go code, when you want
versioned and reviewable prompt files, or when you want to A/B test prompt
variants.

```handlebars
---
model: googleai/gemini-flash-latest
tools:
  - getAttractions
  - getFlightInfo
---

{{role "system"}}
You are a friendly trip planning assistant. Help users plan trips by
suggesting attractions and looking up flight information.

{{history}}
```

```go
// With no source option, DefinePromptAgent loads the prompt registered under
// the agent's own name: ./prompts/tripPlanner.prompt.
tripPlanner := genkit.DefinePromptAgent[any](g, "tripPlanner",
	aix.WithSessionStore(store),
)
```

By default `DefinePromptAgent` renders the prompt registered under the agent's
own name, using the prompt's own default input each turn.

To back several agents with one shared prompt, point each at it with
`aix.WithNamedPrompt(name, input)` and give each its own input. The prompt name
need not match the agent name:

```go
// Three agents share ./prompts/chat.prompt; ChatInput matches its input schema.
for _, p := range []struct{ name, persona string }{
	{"pirate", "a sarcastic pirate"},
	{"chef", "a Michelin-starred chef"},
} {
	genkit.DefinePromptAgent(g, p.name,
		aix.WithNamedPrompt[any]("chat", ChatInput{Personality: p.persona}),
		aix.WithSessionStore(store),
	)
}
```

For full control over orchestration - multiple model calls per turn, custom
progress streaming, and direct session manipulation - see
[Advanced: custom orchestration](#advanced-custom-orchestration) near the end
of this page. Most agents never need it; reach for `DefineAgent` first.

### Typed custom state

To track structured application state alongside the conversation, define a Go
struct and use it as the agent's state type. Genkit infers a JSON schema from
it (used by the developer UI to view and edit state) and threads a typed
`Session` through your tools:

```go
type Task struct {
	ID    int    `json:"id"`
	Title string `json:"title"`
	Done  bool   `json:"done"`
}

type TaskState struct {
	Tasks  []Task `json:"tasks"`
	NextID int    `json:"nextId"`
}

// The state type is inferred from the store's type parameter.
store := localstore.NewInMemorySessionStore[TaskState]()

taskAgent := genkit.DefineAgent(g, "taskAgent",
	aix.InlinePrompt{
		ai.WithSystem("You are a task management assistant. Use the provided tools."),
		ai.WithTools(addTask, toggleTask, removeTask),
	},
	aix.WithSessionStore(store),
)
```

Tools read and mutate this state through the session in their context; see
[Custom state](#custom-state).

## The agent invocation API

In Go you reach an agent two ways, and they look different from each other.
**In-process**, you call the agent's typed methods directly. **Across a
process boundary** (a browser, another service), you [serve the agent over
HTTP](#serving-agents-over-http) and call it from the JavaScript client SDK or
any HTTP client. This section covers the in-process methods, which power CLIs
and embedded apps.

There are three entry points, from simplest to most capable:

```go
// One-shot, text in and out.
out, err := weatherAgent.RunText(ctx, "What is the weather in Tokyo?")
fmt.Println(out.Message.Text())   // final text
fmt.Println(out.FinishReason)     // why the turn ended
fmt.Println(out.SnapshotID)       // snapshot id (server-managed agents)

// One-shot, full control over the input (resume payloads, detach, raw message).
out, err = weatherAgent.Run(ctx, &aix.AgentInput{Message: ai.NewUserTextMessage("...")})

// Streaming and/or multi-turn: open a connection and drive it.
conn, err := weatherAgent.Connect(ctx)
```

`RunText` and `Run` send one turn and block for the result. `Connect`
returns an `*aix.AgentConnection` you can stream from and send follow-up turns
to (covered in [Multi-turn streaming chat](#multi-turn-streaming-chat)). The
connection carries session continuity across turns for you.

The core types:

| Type | What it is | Key members |
|---|---|---|
| `*aix.Agent[State]` | The agent itself | `RunText()`, `Run()`, `Connect()`, `GetSnapshot()`, `GetLatestSnapshot()`, `AbortSnapshot()`, `Store()`, `Name()` |
| `*aix.AgentConnection[State]` | An active invocation (one or many turns) | `SendText()`, `Send()`, `SendResume()`, `Detach()`, `Receive()`, `Output()`, `Custom()`, `Close()` |
| `*aix.AgentInput` | One turn's input | `Message`, `Resume`, `Detach` |
| `*aix.AgentOutput[State]` | A resolved invocation | `Message`, `FinishReason`, `SnapshotID`, `SessionID`, `State`, `Artifacts`, `Error` |
| `*aix.AgentStreamChunk` | One streamed chunk | `ModelChunk`, `CustomPatch`, `Artifact`, `TurnEnd` |
| `*aix.TurnEnd` | End-of-turn signal | `FinishReason`, `SnapshotID` |
| `*aix.SessionSnapshot[State]` | A persisted checkpoint | `SnapshotID`, `SessionID`, `Status`, `State`, `FinishReason` |
| `*aix.SessionState[State]` | Portable conversation state | `Messages`, `Custom`, `Artifacts`, `SessionID` |

**Invocation options.** Calling an entry point with no options starts a fresh
conversation. To continue or branch one, pass an option (they pick the
conversation source):

| Option | Use it to |
|---|---|
| `aix.WithState(state)` | Seed or continue **client-managed** state (the full state blob) |
| `aix.WithSnapshotID(id)` | Resume a specific snapshot (**server-managed**; see [Session branching](#session-branching)) |
| `aix.WithSessionID(id)` | Resume a session's latest snapshot (**server-managed**) |

## State management strategies

Every agent uses one of two strategies for maintaining session state across
turns. This choice affects how a turn is continued, so pick a strategy before
building your client.

| | Server-managed | Client-managed |
|---|---|---|
| **Setup** | Configure a store with `aix.WithSessionStore` | Omit the store |
| **How a turn continues** | Resume with `WithSnapshotID` or `WithSessionID` | Round-trip the full `State` via `WithState` |
| **Server stores state?** | Yes | No |
| **Supports branching?** | Yes | Manual (open multiple connections from the same state) |
| **Supports background execution?** | Yes | No |

In both cases an `*AgentConnection` carries continuity automatically within a
single connection; the difference is *where* the state lives between
connections. See [Session persistence](#session-persistence) for detailed
examples of each.

## Serving agents over HTTP

To reach an agent from another process or a browser, expose it as HTTP
endpoints. The `genkitx` package lays out a default route set for you. Range
over it and mount each route onto a standard `net/http` mux (or any router):

```go
mux := http.NewServeMux()

// One agent's routes...
for _, route := range genkitx.AgentRoutes(weatherAgent) {
	mux.HandleFunc(route.Pattern(), route.Handler())
}

// ...or every registered agent at once.
for _, route := range genkitx.AllAgentRoutes(g) {
	mux.HandleFunc(route.Pattern(), route.Handler())
}

log.Fatal(server.Start(ctx, "127.0.0.1:8080", mux))
```

The route set mirrors what each agent can do:

| Route | When it's present |
|---|---|
| `POST /agents/{name}` | Always. The agent itself, one turn per request (add `?stream=true` for SSE) |
| `POST /agents/{name}/getSnapshot` | Only for **server-managed** agents (a store is configured). Reads a snapshot by ID for restore/inspect |
| `POST /agents/{name}/abortSnapshot` | Only when the store also supports aborting. Cancels in-flight [background work](#background-execution) |

Each route is a POST taking the `{"data": ...}` envelope and returning
`{"result": ...}`, the same convention as the rest of Genkit's reflection API.
`route.Pattern()` is its `"METHOD /path"` and `route.Handler()` builds the
handler; pass `genkit.HandlerOption` values (for example, context providers for
auth) to `Handler()` to apply them per route.

### Calling a Go agent from the JavaScript client SDK

Because the routes speak Genkit's standard wire envelope, the JavaScript agent
client (`remoteAgent` from `genkit/beta/client`) talks to a deployed Go agent
with no server-specific code. Point it at the agent's URL. The only Go-specific
detail is the companion paths: Go serves them at `/getSnapshot` and
`/abortSnapshot`, so override the client's defaults (`/state` and `/abort`):

```ts
import { remoteAgent } from 'genkit/beta/client';

const agent = remoteAgent({
  url: 'http://localhost:8080/agents/weatherAgent',
  // Go's companion routes differ from the JS client's defaults:
  stateUrl: 'http://localhost:8080/agents/weatherAgent/getSnapshot',
  abortUrl: 'http://localhost:8080/agents/weatherAgent/abortSnapshot',
});

const chat = agent.chat();
const res = await chat.send('Weather in Tokyo?');
console.log(res.text);
```

From here the client behaves exactly as in the [JavaScript agents
guide](/docs/agents?lang=js): `chat.send`, `chat.sendStream`, `chat.resume`,
`loadChat`, `detach`, and so on, all backed by your Go agent.

You can also drive the endpoints with any HTTP client. Start a conversation
(no init starts a fresh session); the response carries a `sessionId` you send
back to continue it:

```bash
# Start a conversation.
curl -X POST http://localhost:8080/agents/weatherAgent \
  -H "Content-Type: application/json" \
  -d '{"data": {"message": {"role": "user", "content": [{"text": "Weather in Tokyo?"}]}}}'

# Continue it (server-managed), using the sessionId from the response.
curl -X POST http://localhost:8080/agents/weatherAgent \
  -H "Content-Type: application/json" \
  -d '{"data": {"message": {"role": "user", "content": [{"text": "And Paris?"}]}}, "init": {"sessionId": "SESSION_ID"}}'

# Stream a turn as server-sent events.
curl -N -X POST 'http://localhost:8080/agents/weatherAgent?stream=true' \
  -H "Content-Type: application/json" \
  -d '{"data": {"message": {"role": "user", "content": [{"text": "Weather in London?"}]}}}'
```

The `init` field is the wire counterpart of the invocation options:
`{"sessionId": ...}` / `{"snapshotId": ...}` for server-managed agents, or
`{"state": ...}` for client-managed ones.

## Multi-turn streaming chat

The most common in-process pattern: a streaming chat that sends messages,
receives streamed tokens and tool-call events, and keeps session continuity
across turns. Open a connection with `Connect`, then for each turn send
input and range over `conn.Receive()` until a `TurnEnd` chunk:

```go
conn, err := weatherAgent.Connect(ctx)
if err != nil {
	log.Fatal(err)
}

// Turn 1.
if err := conn.SendText("Weather in London?"); err != nil {
	log.Fatal(err)
}
streamTurn(conn)

// Turn 2 - the same connection carries state forward.
if err := conn.SendText("What about Paris?"); err != nil {
	log.Fatal(err)
}
streamTurn(conn)

// Finalize: closes the input side and returns the resolved invocation.
out, err := conn.Output()

// streamTurn prints one turn's model text and returns at the turn boundary.
func streamTurn(conn *aix.AgentConnection[any]) {
	for chunk, err := range conn.Receive() {
		if err != nil {
			log.Fatal(err)
		}
		if chunk.ModelChunk != nil {
			fmt.Print(chunk.ModelChunk.Text()) // streamed model text
		}
		if chunk.TurnEnd != nil {
			// Turn done. finishReason and snapshotId ride the TurnEnd, so you
			// learn how the turn ended without scanning the message content.
			fmt.Printf("\n[turn: %s]\n", chunk.TurnEnd.FinishReason)
			return
		}
	}
}
```

Breaking out of `Receive()` does not cancel the connection; multi-turn callers
routinely return at the `TurnEnd`, send the next input, and range `Receive()`
again. Call `conn.Output()` to finish the invocation, or cancel the context
passed to `Connect` to abort it.

`chunk.ModelChunk` is the raw model response chunk: `chunk.ModelChunk.Text()`
gives streamed text, and tool requests and responses are in its `Content`
parts. For a non-streaming turn, use `RunText` / `Run` instead.

## Session persistence

This section covers each [state management strategy](#state-management-strategies)
in detail: stores, restoring a conversation, client-managed state, and
branching.

### Server-side stores

A store persists session state across requests. Genkit provides two local
stores; both are generic over the state type:

```go
// File-based persistence: one JSON file per snapshot under dir.
store, err := localstore.NewFileSessionStore[any]("./.genkit/snapshots")

// In-memory (for development and testing).
store := localstore.NewInMemorySessionStore[any]()
```

With a store, each turn produces a snapshot whose ID is on `out.SnapshotID`
and `chunk.TurnEnd.SnapshotID`. A conversation is identified by its
`SessionID` (minted on the first turn, stable across resumes, available on
`out.SessionID`). Track the session ID for a user, and you can restore the
conversation later.

**Resume a conversation.** In-process, resume by session ID (its latest
snapshot) or by a specific snapshot:

```go
// Continue where the session left off.
conn, err := agent.Connect(ctx, aix.WithSessionID[any](sessionID))

// Or resume from an exact snapshot.
conn, err = agent.Connect(ctx, aix.WithSnapshotID[any](snapshotID))
```

**Read a snapshot without starting a turn.** Given an agent reference, read
through the agent to inspect history or render a UI. The agent applies any
configured state transform (a direct `agent.Store()` read returns raw state):

```go
snap, err := agent.GetSnapshot(ctx, snapshotID)
// snap.State.Messages   - full message history
// snap.State.Custom     - custom state
// snap.State.Artifacts  - artifacts
// snap.Status           - pending | completed | aborted | failed | expired

// Or the session's latest snapshot, whatever its status:
latest, err := agent.GetLatestSnapshot(ctx, sessionID)
```

Over HTTP, the same data is reachable through the `getSnapshot` companion
route (and the JavaScript client's `loadChat` / `getSnapshot`).

### Client-managed state

If you don't configure a store, the agent is stateless on the server and the
caller owns the session state. Each result carries the full state on
`out.State`; send it back on the next turn with `WithState`:

```go
// First turn: no store, no init.
out, err := agent.RunText(ctx, "Weather in Tokyo?")

// out.State is the full SessionState (messages + custom + artifacts), with the
// conversation identity inside it. Send it back to continue.
out, err = agent.RunText(ctx, "What about Paris?", aix.WithState(out.State))
fmt.Println(out.State.Custom) // latest custom state
```

The trade-off: no server storage, but the caller holds (and, over HTTP, the
transport sends) the full history every turn. This works well for short
conversations or a fully stateless server.

### Session branching

With a store, you can branch conversations from any snapshot, creating parallel
threads from the same starting point. Open multiple connections from the same
snapshot ID:

```go
out, _ := agent.RunText(ctx, "Hello!")
branchPoint := out.SnapshotID

// Two independent branches from the same point.
out, _ = agent.RunText(ctx, "My name is Alice.", aix.WithSnapshotID[any](branchPoint))
out, _ = agent.RunText(ctx, "My name is Bob.", aix.WithSnapshotID[any](branchPoint))
```

A snapshot is an immutable checkpoint: like a git commit, the original doesn't
change when you branch from it. Each branch creates new, independent snapshots.

### Redacting state and stream output

`WithStateTransform` rewrites session state on its way out to a client without
touching what the server keeps. It runs on `GetSnapshot` / `GetLatestSnapshot`
reads (and the `getSnapshot` companion route), on a client-managed agent's
`out.State`, and on the streamed `CustomPatch` diffs, so redaction reaches every
client-facing surface. The persisted snapshot and the state your agent function
sees stay raw, so the conversation still resumes faithfully.

```go
chatAgent := genkit.DefineAgent(g, "chat",
    aix.InlinePrompt{ai.WithModelName("googleai/gemini-flash-latest")},
    aix.WithSessionStore(store),
    aix.WithStateTransform[MyState](func(ctx context.Context, s *aix.SessionState[MyState]) (*aix.SessionState[MyState], error) {
        return redactPII(ctx, s) // ctx carries the caller's identity for RBAC-aware redaction
    }),
)
```

`WithStreamTransform` is the stream-side counterpart: it rewrites each
`AgentStreamChunk` (model tokens, artifacts, custom patches, and turn-end
signals) as it leaves for the client. A chunk carries no state type, so unlike
`WithStateTransform` its `State` argument cannot be inferred and is passed
explicitly:

```go
    aix.WithStreamTransform[MyState](func(ctx context.Context, c *aix.AgentStreamChunk) (*aix.AgentStreamChunk, error) {
        if c.ModelChunk != nil {
            c.ModelChunk = redactModelChunk(c.ModelChunk)
        }
        return c, nil
    }),
```

Both transforms receive a fresh deep copy they own, and the return value decides
what the client sees:

- Return the (possibly mutated) value with a `nil` error to forward it.
- Return a `nil` value to omit that state, or drop that chunk, from the client's
  view. This is wire-only: the side effects, the persisted snapshot, and the
  final `AgentOutput` still hold the underlying data.
- Return a non-nil error to **fail closed**: the read or invocation fails with
  that error instead of leaking unredacted data, and the error's status code
  (such as `PERMISSION_DENIED`) propagates to the caller.

To redact custom state specifically, prefer `WithStateTransform`: the runtime
applies it before diffing, so the streamed `CustomPatch` deltas stay internally
consistent.

## Custom state and artifacts

### Custom state

Agents don't just produce text; they produce **structured app state** that
drives your UI: exact IDs, counters, structured records, anything the model
can't reliably maintain in chat history alone. Tools read and mutate the typed
state through the `Session` carried in their context, retrieved with
`aix.SessionFromContext` (using the `TaskState` from
[Typed custom state](#typed-custom-state) above).

To **mutate** state, call `session.UpdateCustom` with a function that takes
the current state and returns the new state:

```go
addTask := genkit.DefineTool(g, "addTask",
	"Add a new task to the task list.",
	func(ctx *ai.ToolContext, in struct {
		Title string `json:"title"`
	}) (Task, error) {
		sess := aix.SessionFromContext[TaskState](ctx.Context)
		var created Task
		sess.UpdateCustom(func(s TaskState) TaskState {
			// On a fresh session the state is the zero value, so default
			// per field rather than assuming it's populated.
			if s.NextID == 0 {
				s.NextID = 1
			}
			created = Task{ID: s.NextID, Title: in.Title}
			s.Tasks = append(s.Tasks, created)
			s.NextID++
			return s
		})
		return created, nil
	},
)
```

To **read** state without mutating it, call `session.Custom`:

```go
listTasks := genkit.DefineTool(g, "listTasks", "List all tasks.",
	func(ctx *ai.ToolContext, _ struct{}) ([]Task, error) {
		sess := aix.SessionFromContext[TaskState](ctx.Context)
		return sess.Custom().Tasks, nil
	},
)
```

After a turn, read the final custom state where the strategy keeps it: from
the snapshot for server-managed agents, or `out.State.Custom` for
client-managed ones.

```go
// Server-managed: read it back from the snapshot.
out, _ := taskAgent.RunText(ctx, "Add: buy groceries")
snap, _ := taskAgent.GetSnapshot(ctx, out.SnapshotID)
fmt.Println(snap.State.Custom.Tasks)
```

When **streaming**, the connection tracks custom state live. Each mutation
emits a `CustomPatch` chunk, which the connection applies to an internal copy
exposed by `conn.Custom()`, so you can render progress as it changes:

```go
for chunk, err := range conn.Receive() {
	if err != nil {
		log.Fatal(err)
	}
	if len(chunk.CustomPatch) > 0 {
		state, _ := conn.Custom() // live custom state after this patch
		updateProgressUI(state)
	} else if chunk.ModelChunk != nil {
		fmt.Print(chunk.ModelChunk.Text())
	}
}
```

### Artifacts

Tools can produce artifacts (generated files, code, images) by adding them to
the session:

```go
generateFile := genkit.DefineTool(g, "generateFile",
	"Add a generated code file to the workspace.",
	func(ctx *ai.ToolContext, in struct {
		Name    string `json:"name"`
		Content string `json:"content"`
	}) (string, error) {
		sess := aix.SessionFromContext[any](ctx.Context)
		sess.AddArtifacts(&aix.Artifact{
			Name:  in.Name,
			Parts: []*ai.Part{ai.NewTextPart(in.Content)},
		})
		return "Artifact " + in.Name + " added.", nil
	},
)
```

Artifacts are available on `out.Artifacts`, stream in as `chunk.Artifact`, and
are persisted in the session state. A [custom agent](#advanced-custom-orchestration)
can also stream one directly with `resp.SendArtifact`.

## Agents with interrupts

Agents work with [interrupts](/docs/interrupts) for human-in-the-loop
workflows: approvals, confirmations, or any time you need user input mid-turn.
Define an **interruptible tool** with `genkit.DefineInterruptibleTool`. Its
function takes a third parameter, a pointer to the resume payload, that is
`nil` on the first call and populated when the caller resumes. Inside, call
`tool.Interrupt` to pause and hand typed data to the caller:

```go
type TransferInput struct {
	ToAccount string  `json:"toAccount"`
	Amount    float64 `json:"amount"`
}

// Confirmation is the resume payload; it arrives as the tool's third
// parameter when the caller resumes.
type Confirmation struct {
	Approved bool `json:"approved"`
}

// ApprovalRequest is the typed data the tool hands the caller when it pauses.
type ApprovalRequest struct {
	Message string  `json:"message"`
	Amount  float64 `json:"amount"`
}

transferMoney := genkit.DefineInterruptibleTool(g, "transferMoney",
	"Transfer money to a specified account.",
	func(ctx context.Context, in TransferInput, confirm *Confirmation) (string, error) {
		// First pass and a large transfer: pause for approval.
		if confirm == nil && in.Amount > 100 {
			return "", tool.Interrupt(ApprovalRequest{
				Message: fmt.Sprintf("Transfer $%.2f to %s. Approve?", in.Amount, in.ToAccount),
				Amount:  in.Amount,
			})
		}
		// Resumed with a decision.
		if confirm != nil && !confirm.Approved {
			return "Transfer cancelled.", nil
		}
		return fmt.Sprintf("Transferred $%.2f to %s.", in.Amount, in.ToAccount), nil
	},
)

bankingAgent := genkit.DefineAgent[any](g, "bankingAgent",
	aix.InlinePrompt{
		ai.WithSystem("You are a banking assistant. Use transferMoney to move money."),
		ai.WithTools(transferMoney),
	},
	aix.WithSessionStore(store),
)
```

When a turn pauses, the model's chunks carry interrupted tool requests
(`chunk.ModelChunk.Interrupts()`) and the turn ends with finish reason
`aix.AgentFinishReasonInterrupted`. Collect the interrupts, get the user's
decision, and resume with `conn.SendResume`:

```go
conn, _ := bankingAgent.Connect(ctx)
conn.SendText("Transfer $500 to my savings account")

var interrupts []*ai.Part
for chunk, err := range conn.Receive() {
	if err != nil {
		log.Fatal(err)
	}
	if chunk.ModelChunk != nil {
		fmt.Print(chunk.ModelChunk.Text())
		interrupts = append(interrupts, chunk.ModelChunk.Interrupts()...)
	}
	if chunk.TurnEnd != nil {
		break
	}
}

if len(interrupts) > 0 {
	resume := &aix.ToolResume{}
	for _, part := range interrupts {
		// Read the typed request and decide. Re-run the tool with the
		// decision (the "restart" pattern).
		req, _ := tool.InterruptAs[ApprovalRequest](part)
		approved := askUser(req.Message) // your UI
		p, err := tool.Resume(part, Confirmation{Approved: approved})
		if err != nil {
			log.Fatal(err)
		}
		resume.Restart = append(resume.Restart, p)
	}
	conn.SendResume(resume)

	// The resumed turn streams on the next Receive; consume it.
	for chunk, err := range conn.Receive() {
		if err != nil {
			log.Fatal(err)
		}
		if chunk.ModelChunk != nil {
			fmt.Print(chunk.ModelChunk.Text())
		}
		if chunk.TurnEnd != nil {
			break
		}
	}
}
```

Genkit supports two resumption patterns, sorted into the two halves of
`aix.ToolResume`:

| | Respond | Restart |
|---|---|---|
| **When to use** | You supply the tool's answer directly | The tool re-executes with the resume payload |
| **Tool runs again?** | No | Yes (with the typed `*Resume` populated) |
| **Build the part with** | `tool.Respond(part, output)` | `tool.Resume(part, data)` |
| **Goes in** | `ToolResume.Respond` | `ToolResume.Restart` |

The restart pattern is **more secure** for approval gates: the resume payload
can only be set by the caller's resume, never by the model, so the model can't
forge approval through prompt injection. Keep the approval check in the tool
body (which runs on resume) rather than trusting the model to gate itself.

Note that arbitrary time can pass between an interrupt and its resumption. With
a store, all session state is persisted, so the resumption can be a brand-new
connection that resumes from the snapshot (`WithSnapshotID` / `WithSessionID`),
not necessarily the same live connection.

## Background execution

For long-running tasks (report generation, deep research), detach the
invocation: the caller submits a turn, gets a pending snapshot ID immediately,
and the agent keeps working in the background. This requires a **server-managed
agent** (a store). Send a turn with `Detach` set, then read the pending ID off
the output:

```go
conn, _ := backgroundAgent.Connect(ctx)
conn.Send(&aix.AgentInput{
	Detach:  true,
	Message: ai.NewUserTextMessage("Write a report on AI trends."),
})

out, _ := conn.Output()
pendingID := out.SnapshotID // available right away; FinishReason is "detached"
```

**Poll for completion.** If the store supports status subscriptions (the
`SnapshotSubscriber` capability, which `localstore` implements), subscribe to
status changes; otherwise read the snapshot on an interval:

```go
store := backgroundAgent.Store()

if sub, ok := store.(aix.SnapshotSubscriber); ok {
	for status := range sub.OnSnapshotStatusChange(ctx, pendingID) {
		if status != aix.SnapshotStatusPending {
			break
		}
	}
}

final, _ := backgroundAgent.GetSnapshot(ctx, pendingID)
// final.Status is completed | failed | aborted (expired if the worker died);
// final.State holds the result.
```

**Abort** the background work through the agent:

```go
backgroundAgent.AbortSnapshot(ctx, pendingID)
```

Over HTTP, the `getSnapshot` and `abortSnapshot` companion routes expose the
same poll-and-abort flow to remote clients (and to the JavaScript client's
`detach` / `poll` / `abort`).

## Multi-agent delegation

Genkit Go has no dedicated multi-agent middleware yet. The idiomatic pattern is
to wrap a specialized sub-agent in a **tool**, since an agent is just an action
you can run. The orchestrator's model calls the tool, the tool runs the
sub-agent and returns its answer, and delegation flows through the normal
tool-calling loop:

```go
// A specialized sub-agent.
researcher := genkit.DefineAgent[any](g, "researcher",
	aix.InlinePrompt{
		ai.WithSystem("You are a thorough research assistant."),
		ai.WithTools(getWebResults),
	},
	aix.WithDescription[any]("Finds well-sourced answers."),
)

// Expose it to the orchestrator as a tool.
askResearcher := genkit.DefineTool(g, "askResearcher",
	"Delegate a research question to the research assistant.",
	func(ctx *ai.ToolContext, in struct {
		Question string `json:"question"`
	}) (string, error) {
		out, err := researcher.RunText(ctx.Context, in.Question)
		if err != nil {
			return "", err
		}
		return out.Message.Text(), nil
	},
)

// The orchestrator delegates by calling the tool.
orchestrator := genkit.DefineAgent[any](g, "orchestrator",
	aix.InlinePrompt{
		ai.WithSystem("Analyze the request, delegate to a specialist when useful, then synthesize a final answer."),
		ai.WithTools(askResearcher),
	},
)
```

The `askResearcher` tool calls appear in the orchestrator's stream like any
other tool, so a client can show delegation status. Each delegated call is a
fresh sub-invocation of `researcher` (stateless by default); pass invocation
options to `RunText` if you want the sub-agent to share or persist a session.
This gives you explicit control over what the orchestrator forwards and how
many times it can delegate.

## Advanced: custom orchestration

Most agents are well served by `genkit.DefineAgent`, which runs one model call
with an automatic tool-calling loop. When you need full control over a turn,
use `genkit.DefineCustomAgent`. Your function receives a `Responder` for
streaming output and a `SessionRunner` for turn and state management; call
`sess.Run` to enter the per-turn loop. Inside, you can:

- **Chain multiple model calls** with custom logic between them.
- **Stream human-readable progress** by mutating custom state with
  `sess.UpdateCustom` (the runtime auto-emits a `CustomPatch` chunk).
- **Manipulate the session directly** with `sess.AddMessages` and
  `sess.UpdateCustom`, controlling exactly what gets persisted.
- **Use different models for different steps.**

Even with full control, the framework still owns session state, snapshot
writes, and the detach lifecycle. This research agent decomposes a question,
researches each part, and streams status throughout:

```go
type ResearchState struct {
	Status       string   `json:"status"`
	SubQuestions []string `json:"subQuestions"`
}

researchAgent := genkit.DefineCustomAgent(g, "researchAgent",
	func(ctx context.Context, resp aix.Responder, sess *aix.SessionRunner[ResearchState]) (*aix.AgentResult, error) {
		var last *ai.Message
		err := sess.Run(ctx, func(ctx context.Context, input *aix.AgentInput) (*aix.TurnResult, error) {
			question := input.Message.Text()

			// Step 1: decompose with a fast model. Mutating custom state
			// auto-streams a CustomPatch chunk to the client.
			sess.UpdateCustom(func(s ResearchState) ResearchState {
				s.Status = "Decomposing question..."
				return s
			})
			decompose, err := genkit.Generate(ctx, g,
				ai.WithModelName("googleai/gemini-flash-lite-latest"),
				ai.WithPrompt("Break this into 2-3 sub-questions, one per line: "+question),
			)
			if err != nil {
				return nil, err
			}
			subQuestions := strings.Split(strings.TrimSpace(decompose.Text()), "\n")

			// Step 2: research each sub-question (status updates each time).
			var findings strings.Builder
			for i, q := range subQuestions {
				sess.UpdateCustom(func(s ResearchState) ResearchState {
					s.Status = fmt.Sprintf("Researching (%d/%d)...", i+1, len(subQuestions))
					return s
				})
				answer, err := genkit.GenerateText(ctx, g, ai.WithPrompt("Answer concisely: "+q))
				if err != nil {
					return nil, err
				}
				fmt.Fprintf(&findings, "Q: %s\nA: %s\n\n", q, answer)
			}

			// Step 3: synthesize, streaming the final answer token by token.
			sess.UpdateCustom(func(s ResearchState) ResearchState {
				s.Status = "Synthesizing..."
				return s
			})
			for v, err := range genkit.GenerateStream(ctx, g,
				ai.WithPrompt("Synthesize these findings into a cohesive answer:\n"+findings.String()),
			) {
				if err != nil {
					return nil, err
				}
				if v.Done {
					last = v.Response.Message
					sess.AddMessages(last) // persist the final message in history
					return &aix.TurnResult{FinishReason: aix.AgentFinishReason(v.Response.FinishReason)}, nil
				}
				resp.SendModelChunk(v.Chunk) // forward each model chunk to the client
			}
			return nil, nil
		})
		if err != nil {
			return nil, err
		}
		return &aix.AgentResult{Message: last}, nil
	},
)
```

To stream human-readable progress, mutate custom state with `sess.UpdateCustom`
as shown (the client receives it on `chunk.CustomPatch` / `conn.Custom()`).
`resp.SendModelChunk` forwards model tokens, and `resp.SendArtifact` streams an
artifact.

## Error handling

Go agents distinguish two failure tiers, and they surface differently.

An **init failure** (an unknown session or snapshot ID, state sent to a
store-backed agent, a cancelled context) means the invocation never started.
It comes back as a returned `error`:

```go
out, err := agent.RunText(ctx, userMessage)
if err != nil {
	// The invocation never produced a turn.
	log.Fatal(err)
}
```

An **in-band failure** (a turn whose model or tool call errored) does *not*
return an error. It resolves as a failed `AgentOutput`, leaving the session at
its last-good state so the conversation stays recoverable:

```go
if out.FinishReason == aix.AgentFinishReasonFailed {
	fmt.Println("agent failed:", out.Error.Status, out.Error.Message)

	// Resume point: the last committed turn (server-managed), or the
	// last-good state on out.State (client-managed). The next turn from here
	// continues the conversation without losing it.
	fmt.Println("resume from snapshot:", out.SnapshotID)
	return
}
fmt.Println(out.Message.Text())
```

With `Connect`, the same failure surfaces as `TurnEnd.FinishReason ==
aix.AgentFinishReasonFailed` during streaming, and on `conn.Output()` after.

### When a tool returns an error

A tool that returns an error aborts the turn the same way any other failure
does: it resolves as a failed `AgentOutput`, and the session is left at its
last-good state. The error is **not** fed back to the model, so the model won't
see it or try to recover on its own.

If instead you want the model to react to a failure (retry with different
arguments, tell the user an item wasn't found), don't return an error. Return a
normal result that describes the outcome so the model can act on it:

```go
type CheckInResult struct {
	OK    bool   `json:"ok"`
	Error string `json:"error,omitempty"`
}

checkInHabit := genkit.DefineTool(g, "checkInHabit",
	"Mark a habit as completed for today.",
	func(ctx *ai.ToolContext, in struct {
		ID int `json:"id"`
	}) (CheckInResult, error) {
		habit := findHabit(in.ID)
		if habit == nil {
			// The model sees this and can tell the user the id wasn't found.
			return CheckInResult{OK: false, Error: fmt.Sprintf("No habit with id %d.", in.ID)}, nil
		}
		habit.CheckedInToday = true
		return CheckInResult{OK: true}, nil
	},
)
```

Reserve a returned error for genuinely unrecoverable conditions where you want
the whole turn to fail; model expected, recoverable outcomes as tool output.
